# 1. 概述

## 1.1 项目背景

随着 Gentle Monster 中国区售后业务发展，售后流程已覆盖维修、配件、咨询、支付、物流、回访等多个环节，并涉及小程序、SAP、物流、短信等系统协同。随着业务链路和参与对象增加，现有管理方式对流程统一、状态透明、系统协同和过程追踪提出了更高要求。

因此，需要建设一套覆盖售后全流程的管理系统，统一承接售后工单、维修处理、配件库存协同及客户服务相关业务，打通相关系统数据，形成清晰、可控、可追踪的业务闭环。

## 1.2 项目目标

本项目旨在构建一套覆盖售后全流程的管理系统，作为客户服务的核心平台，支撑售后业务规范、高效运行。

具体目标如下：

1. 统一售后流程，规范申请、检测、判定、维修、支付、出库、履约、回访等业务环节。  
2. 建立统一状态管理机制，提高流程透明度和业务可控性。  
3. 打通与小程序、SAP、物流等系统的数据协同，提升处理效率。  
4. 支撑维修、配件、库存、咨询、问卷等业务的统一管理。  
5. 实现关键过程留痕，便于查询、追踪和问题定位。  


# 2. 业务范围与系统边界

## 2.1 业务范围

系统覆盖售后服务完整流程：

申请 → 受理 → 收货 → 检测 → 判定 → 处理 → 出库 → 履约 → 完成

支持以下业务场景：

- 维修（有偿 / 无偿）
- 产品更换（同品 / 他品）
- 合作厂家 / 3PL 处理
- 无法修理 / 假货处理
- 异常处理（支付 / 库存 / 物流等）
- 服务完成后的问卷与回访



## 2.2 系统边界

| 业务环节 | 责任系统          | 说明           |
| -------- | ----------------- | -------------- |
| 申请     | 小程序 / 门店     | 用户或门店发起 |
| 流程控制 | PS Admin          | 核心系统       |
| 支付     | 小程序 + 支付系统 | 完成支付       |
| 库存     | SAP               | 库存与出入库   |
| 物流     | 物流系统          | 配送与签收     |

说明：

- PS Admin 负责流程驱动与状态管理
- 外部系统负责具体业务执行
- 各系统通过接口进行数据协同


## 2.3 系统设计原则

### 2.3.1 所有业务以工单为核心
系统中的申请、受理、维修、支付、出库、完成及异常处理等业务，均以工单作为统一管理载体。工单作为业务处理主线，承载流程状态、业务数据及处理结果，确保全过程信息统一归集。

### 2.3.2 流程由状态驱动推进
系统流程通过状态进行驱动和控制，原则上按既定流转顺序逐步推进。各环节需在满足对应条件后进入下一状态，确保流程执行规范、有序，避免出现随意跳转、遗漏环节或状态混乱的情况。

### 2.3.3 系统间职责明确分离
各系统按照职责边界承担相应功能。PS Admin 负责工单管理、流程驱动、状态控制及业务数据整合；外部系统负责支付、库存、物流等具体业务执行。各系统之间通过接口进行数据交互，确保职责清晰、协同有序。


## 2.4 多语言要求

系统需支持中文、英文、韩文三种语言切换。

## 3.2 流程说明

整体售后流程如下：

客户通过小程序或门店发起售后申请，系统生成售后工单，并依次进入收货、检测、判定、处理、履约及完成阶段。

根据判定结果，工单可进入维修、换货、合作厂家 / 3PL 处理，或直接结束（如无法修理、假货等）等不同路径；其中有偿维修需先完成支付后再进入后续处理。

处理完成后，系统同步相关结果，并通过快递返还或门店取件完成交付。服务完成后，系统可继续触发满意度调查、问卷或回访流程。

流程主要分为以下阶段：

1）申请  
客户通过小程序或门店发起售后申请，系统生成工单。

2）收货与检测  
产品完成寄送或接收后，进入收货、入库及检测环节。

3）判定与处理  
系统根据检测结果进入维修、换货、外部处理或结束流程；如为有偿维修，需先完成支付。

4）履约  
处理完成后进入出库与交付环节，通过快递返还或门店取件完成履约。

5）完成  
客户签收或完成取件后，工单进入服务完成，并可进入满意度调查、问卷或回访流程。

系统在关键流程节点（如接收完成、判定结果、支付状态、履约完成等）将同步触发客户通知，用于告知服务进度及结果。
通知支持系统自动触发及人工触发两种方式，具体以业务配置为准。


## 3.3 关键分支流程

### 3.3.1 申请流程

客户可通过以下两种方式发起售后申请：

小程序申请（客户自行提交）
门店申请（门店代客提交）

申请后系统生成工单，并进入收货流程。

说明：

所有申请统一进入同一套流程管理
门店与小程序仅作为入口差异，不影响后续流程
客户在还未进入维修流程前都可以撤销申请


```mermaid
flowchart TD

A1[用户在小程序发起申请] --> A2[提交售后信息]
A2 --> A3[生成工单（RECEIVED）]
A3 --> A4[用户填写寄件信息]
A4 --> A5[等待收货（WAIT_RECEIVE）]

B1[客户到门店申请] --> B2[门店创建工单]
B2 --> B3[生成工单（RECEIVED）]
B3 --> B4[门店代寄或客户寄件]
B4 --> B5[等待收货（WAIT_RECEIVE）]

A5 --> C1[维修中心签收]
B5 --> C1
C1 --> C2[入库完成（RECEIVED_WAREHOUSE）]
C2 --> C3[进入检测（INSPECTING）]
```

说明：

- 小程序与门店为两个申请入口，但统一由 PS Admin 生成工单
    
- 进入检测前必须完成签收与入库
    
- 申请流程结束后，工单进入检测主流程

客户可通过两种方式提交产品：

快递寄送至维修中心
门店收件后转寄

产品到达后完成收货登记及入库，并进入检测阶段。

若收货过程中出现异常（如破损、信息不一致等），需记录并处理。

### 3.3.3 检测与判定流程

产品检测完成后，进入判定阶段，根据检测结果决定处理方式：

- 无偿维修
- 有偿维修（需支付）
- 产品更换
- 3PL / 合作厂家处理
- 无法修理
- 假货处理
- 咨询确认（需客户决策）

系统将根据判定结果向客户发送对应通知（如有偿维修、无偿维修、产品更换等），以确保客户及时了解处理方案。

```mermaid
flowchart TD

A[检测中（INSPECTING）] --> B[检测完成 → 待判定（WAIT_DECISION）]
B --> C{判定结果}

C --> D[无偿维修（WAIT_REPAIR）]
C --> E[有偿维修 → 生成报价]
E --> F[等待支付（WAIT_PAYMENT）]

C --> G[产品更换处理中（EXCHANGE_PROCESSING）]
C --> H[3PL处理 → 进入维修流程（WAIT_REPAIR）]
C --> I[合作厂家维修 → 进入维修流程（WAIT_REPAIR）]

C --> J[无法修理（UNREPAIRABLE）]
C --> K[假货（FAKE_PRODUCT）]

C --> L[需咨询（WAIT_CUSTOMER_CONFIRM）]
```


### 3.3.4 咨询流程

咨询状态默认值为“咨询待机”；当“咨询完成”后，状态自动返回判定节点。

在以下场景需发起咨询：

- 维修方案需客户确认
    
- 费用变更
    
- 他品交换

- 异常情况说明

- 合作厂家延迟
    
客户确认后流程继续，若客户拒绝或超时，可关闭工单。

```mermaid
flowchart TD

A[待客户确认（WAIT_CUSTOMER_CONFIRM）] --> B[发送咨询通知]
B --> C[客户确认/回复]
C --> D{是否确认}

D -->|是| E[返回判定（WAIT_DECISION）]
D -->|否/超时| F[取消工单（CLOSED_CANCELLED）]
```


### 3.3.5 支付流程

当工单判定为有偿维修场景时，系统进入支付流程。

当判定为有偿维修时：

1. 系统从 SAP 查询维修费用；
2. 系统生成待支付信息，并同步至支付平台；
3. 客户可通过小程序或网页版（Web）完成支付；
4. 支付结果回传 PS Admin，并用于更新工单状态。

系统将根据支付状态（待支付、已支付、超时未支付等）触发相应通知及提醒机制。
其中待支付状态下：
- 在“判定完成”后 +D1 起，每天发送 1 次，共发送 5 次
- 最终未付款提醒，在“判定完成”后 +D30 发送一次
- 超时后关闭工单

补充取消规则：

1. 在“维修进行前”阶段，客户可通过线上系统自行取消售后申请；
2. 若当前工单存在已生成但未完成或需撤销的支付单，取消申请时应同步执行支付取消（含 CancelPayment）；
3. 客户可取消的范围，包含待支付、支付处理中、待维修等尚未进入“维修进行中（REPAIRING）”的阶段；
4. 客户发起取消后，系统需记录取消时间、取消来源、取消原因及处理结果；
5. 若取消成功，工单进入取消关闭状态；若支付取消失败，则需提示人工介入处理。

### 3.3.6 维修执行流程
维修流程包括：

- 进入待维修状态
- 到达时间条件后进入维修中
  
    - 若为 “无偿”：判定完成 +D3 后状态自动变更为 “维修进行中”
    - 若为 “有偿”：付款完成 +D3 后状态自动变更为 “维修进行中”
    - 若为 “3PL”：付款完成 +D1 后状态自动变更为 “维修进行中”

- 执行维修或更换
- 完成后进入待出库

维修过程中可能涉及配件更换与库存扣减。

```mermaid
flowchart TD

A[判定完成（JUDGED）] --> B{维修类型是否有偿}

B -->|无偿| C[等待至 判定完成 +D3]
B -->|有偿| D{是否已付款（PAID）}
B -->|3PL| E{是否已付款（PAID）}

D -->|是| F[等待至 付款完成 +D3]
D -->|否| Z1[保持待维修（WAIT_REPAIR）]

E -->|是| G[等待至 付款完成 +D1]
E -->|否| Z1

C --> H[状态自动变更为 维修进行中（REPAIRING）]
F --> H
G --> H

H --> I[维修人员领取设备]
I --> J[填写维修开始日期]
J --> K[执行维修处理]

K --> L{维修处理类型}
L -->|维修处理| M[执行维修操作]
L -->|产品更换| N[扣减对应配件]
L -->|他品更换| O[扣减新产品库存]

M --> P[记录维修结果]
N --> P
O --> P

P --> Q[维修完成]
Q --> R[待出库（WAIT_OUTBOUND）]
```
    

### 3.3.7 配件与库存处理流程

在维修处理过程中，若涉及配件更换，系统需执行配件选择、库存校验及配件消耗记录等操作。  
该流程作为维修流程中的子流程执行。

1. 配件选择规则  
    维修人员可对系统默认配件进行确认、修改、新增或删除。
    
2. 库存校验规则  
    系统在维修完成前校验配件库存是否满足消耗需求。
    
3. 库存扣减时机  
    配件库存在“维修完成提交时”统一扣减，避免在维修过程中产生库存不一致问题。
    
4. 库存不足处理  
    当库存不足时，系统应支持：
    
    - 触发配件申请流程
    - 或生成库存预警
        
5. 数据记录  
    系统需记录：
    
    - 配件消耗明细
    - 库存变动记录
    - 操作人及时间

```mermaid
D --> E[维修执行]

E --> F[提交维修完成]

J --> K[记录配件消耗明细]

L --> M[维修完成（WAIT_OUTBOUND）]

H -->|否| N{处理方式}

N -->|配件申请| O[触发配件申请流程]
O --> P[等待配件到位]
P --> G

N -->|库存预警| Q[生成库存预警]
Q --> R[人工处理后继续]
R --> G
```

说明：

- 配件与库存流程为维修执行子流程，不单独形成主状态链
    
- 库存扣减统一发生在维修完成提交时
    
- 库存不足时需阻断维修完成提交
    

### 3.3.8 合作厂家 / 3PL 流程

部分工单可由外部执行：

发往合作厂家或3PL
外部完成处理后返回
继续进入出库流程

说明：

- 合作厂家 / 3PL 流程在业务上是外部执行，在系统状态上仍纳入维修主链
- 处理完成后统一返回待出库状态
- 若涉及费用，仍需先经过支付流程
- 当维修执行方为「3PL」时，SAP 负责生成 DO 单据并通知 WMS 发货，出库信息回传至 PS Admin

```mermaid
flowchart TD

A[待维修（WAIT_REPAIR）] --> B[发往合作厂家/3PL]
B --> C[外部处理（REPAIRING）]
C --> D[处理完成]
D --> E[3PL场景：SAP生成DO并通知WMS发货]
E --> F[出库信息回传PS Admin]
F --> G[待出库（WAIT_OUTBOUND）]
```

是否存在销毁或回收流程？

### 3.3.10 出库与履约流程

处理完成后进入交付阶段。系统在出库准备、出库完成、物流发货、配送签收等关键节点，需同步更新小程序及 Web 端状态，并向客户发送履约进度通知。

一、流程说明

- 1）出库准备  
工单处理完成后进入出库准备阶段，系统同步更新状态值及相关数据。

- 2）发货批次  
当工单状态为“维修完成”时，系统于每日下午 3 点启动发货批次任务。未付款或已取消的工单不参与批次处理。

- 3）物流处理  
批次任务执行后，系统生成快递申请、打印运单，并将运单号及相关物流信息回传至 PS Admin。

- 4）手动录入物流信息  
当维修订单状态达到“出库准备完成”后，PS CN 可在当日下午 3 点至次日下午 3 点期间手动录入快递单号及相关物流信息，并同步至小程序。

- 5）自动申请控制  
如在上述时间窗口内已手动录入物流信息，则系统在次日下午 3 点不再自动为该工单生成快递申请。

- 6）履约完成  
快递场景下，配送完成后按工作日 +D1 进入“服务完成”；门店场景下，门店收货完成后按工作日 +D1 进入“服务完成”。

```mermaid
flowchart TD

A[待出库（WAIT_OUTBOUND）] --> B[出库准备完成（OUTBOUND_READY）]
B --> C[同步状态至小程序/Web]

C --> D{是否在手动录入窗口期内}
D -->|是| E[PS CN手动录入物流信息]
D -->|否| F[等待自动批次任务]

E --> G[同步物流信息至小程序/Web]
G --> H[跳过自动快递申请]

F --> I{次日下午3点是否已手动录入}
I -->|是| H
I -->|否| J[系统自动生成快递申请]
J --> K[打印运单]
K --> L[获取物流单号]
L --> M[回传物流单号至PS Admin]
M --> N[同步物流信息至小程序/Web]

H --> O[进入物流配送]
N --> O

O --> P{交付方式}
P -->|快递| Q[配送完成（DELIVERED）]
Q --> R[等待工作日 +D1]
R --> S[服务完成（COMPLETED）]

P -->|门店自提| T[门店收货完成（STORE_ARRIVED）]
T --> U[等待工作日 +D1]
U --> S
```


### 3.3.11 问卷流程

服务完成后，系统生成满意度问卷，并由小程序向客户展示与收集反馈。  
PS Admin 负责接收问卷结果并更新问卷及回访信息。

```mermaid
flowchart TD

A[已完成（COMPLETED）] --> B[生成问卷]
B --> C[小程序发送问卷]
C --> D[用户填写问卷]

D --> E{是否在截止日期前填写}

E -->|是| F[小程序提交问卷结果]
F --> G[PS Admin接收问卷数据]
G --> H[更新问卷状态（SURVEY_REPLIED）]
H --> I[更新回访状态（FOLLOW_UP_COMPLETED）]

E -->|否| J[问卷过期（SURVEY_EXPIRED）]
J --> K[更新回访状态（FOLLOW_UP_CLOSED）]
```

说明：

- 工单进入“服务完成”后，系统自动生成问卷并设置截止日期。
- 问卷由小程序端展示，客户可在截止日期前完成填写。
- 客户提交问卷后，系统记录评分及反馈内容，并更新问卷状态。
- 若客户在截止日期前未填写，问卷自动过期，并关闭回访。
- 问卷与回访数据用于服务质量分析，不影响工单主状态。

### 3.3.12 异常处理流程

系统需支持关键异常处理：

| 异常类型 | 处理方式                     |
| -------- | ---------------------------- |
| 支付失败 | 重新发起或关闭               |
| 支付超时 | 自动关闭工单                 |
| 库存不足 | 阻断流程或触发申请           |
| 物流异常 | 人工介入处理                 |
| 数据异常 | 系统校验与修复               |
| 咨询超时 | 取消关闭或人工跟进           |
| 维修取消 | 客户线上取消或管理员人工取消 |

1. 在“维修进行中（REPAIRING）”至“维修完成”阶段，不支持客户自行取消；
2. 该阶段如需取消，由管理员手动执行取消操作，即 CancelPayment（含支付取消）；
3. 管理员取消时，系统需记录取消原因、取消时间、操作人及处理结果；
4. 若取消时已进入配送准备、已生成出库任务或已进入配送批次，需先确认取消件的配送批次处理逻辑，再执行后续关闭；
5. 若取消后仍需返还产品，应明确后续履约方式及状态流转规则。

```mermaid
flowchart TD

A[任意业务节点] --> B{是否发生异常}

B -->|否| C[继续原流程]

B -->|是| D[记录异常信息]
D --> E[异常分类]

E --> F[物流异常]
E --> G[库存异常]
E --> H[支付异常]
E --> I[数据异常]
E --> J[维修异常]
E --> K[咨询异常]

F --> F1[通知物流/客户]
F1 --> F2[跟踪处理]
F2 --> L[恢复流程]

G --> G1[触发配件申请/库存预警]
G1 --> L

H --> H1[允许重新支付]
H1 --> L

I --> I1[人工修正数据]
I1 --> L

J --> J1[维修中断/重新评估]
J1 --> L

K --> K1[人工跟进或关闭工单]
K1 --> L

L --> M{是否可恢复}

M -->|是| N[返回原流程继续]
M -->|否| O[关闭工单（对应关闭状态）]
```

说明：

- 异常处理流程覆盖主流程各节点
    
- 异常是否可恢复，由具体异常类型和业务规则决定
    
- 无法恢复时需落到明确关闭状态，不允许悬空

## 4.4 通知协同

小程序通知适合用于工单状态更新、支付提醒、维修进度、发货进度及问卷邀请等轻量通知场景。  
但小程序订阅消息通常需由用户主动授权，常见的一次性订阅模式下，授权一次通常仅支持发送一条消息，因此不适合单独承担所有高频提醒场景。对于受理成功、支付催缴、无法维修、发货、到店提醒等关键节点，建议结合短信或邮件进行补充，以保证通知触达效果。

### 4.4.1 各通知渠道定位

| 通知渠道   | 适用场景                                         | 特点                             |
| ---------- | ------------------------------------------------ | -------------------------------- |
| 小程序通知 | 状态更新、支付提醒、维修进度、发货进度、问卷邀请 | 适合轻量提醒，可直接跳转业务页面 |
| 短信通知   | 受理成功、支付催缴、无法维修、发货、到店提醒     | 触达直接，适合关键节点通知       |
