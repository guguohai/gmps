# 1. 概述

## 1.1 项目背景

随着 Gentle Monster 中国区售后业务发展，售后流程已覆盖维修、配件、咨询、支付、物流、回访等多个环节，并涉及小程序、SAP、物流、短信等系统协同。随着业务链路和参与对象增加，现有管理方式对流程统一、状态透明、系统协同和过程追踪提出了更高要求。

因此，需要建设一套覆盖售后全流程的管理系统，统一承接售后工单、维修处理、配件库存协同及客户服务相关业务，打通相关系统数据，形成清晰、可控、可追踪的业务闭环。

## 1.2 项目目标

本项目旨在构建一套覆盖售后全流程的管理系统，作为客户服务的核心平台，支撑售后业务规范、高效运行。

具体目标如下：

1. 统一售后流程，规范申请、检测、判定、维修、支付、出库、履约、回访等业务环节。  
2. 建立统一状态管理机制，提高流程透明度和业务可控性。  
3. 打通与小程序、SAP、物流等系统的数据协同，提升处理效率。  
4. 支撑维修、配件、库存、咨询、问卷等业务的统一管理。  
5. 实现关键过程留痕，便于查询、追踪和问题定位。  


# 2. 业务范围与系统边界

## 2.1 业务范围

系统覆盖售后服务完整流程：

申请 → 受理 → 收货 → 检测 → 判定 → 处理 → 出库 → 履约 → 完成

支持以下业务场景：

- 维修（有偿 / 无偿）
- 产品更换（同品 / 他品）
- 合作厂家 / 3PL 处理
- 无法修理 / 假货处理
- 异常处理（支付 / 库存 / 物流等）
- 服务完成后的问卷与回访



## 2.2 系统边界

| 业务环节 | 责任系统          | 说明           |
| -------- | ----------------- | -------------- |
| 申请     | 小程序 / 门店     | 用户或门店发起 |
| 流程控制 | PS Admin          | 核心系统       |
| 支付     | 小程序 + 支付系统 | 完成支付       |
| 库存     | SAP               | 库存与出入库   |
| 物流     | 物流系统          | 配送与签收     |

说明：

- PS Admin 负责流程驱动与状态管理
- 外部系统负责具体业务执行
- 各系统通过接口进行数据协同


## 2.3 系统设计原则

### 2.3.1 所有业务以工单为核心
系统中的申请、受理、维修、支付、出库、完成及异常处理等业务，均以工单作为统一管理载体。工单作为业务处理主线，承载流程状态、业务数据及处理结果，确保全过程信息统一归集。

### 2.3.2 流程由状态驱动推进
系统流程通过状态进行驱动和控制，原则上按既定流转顺序逐步推进。各环节需在满足对应条件后进入下一状态，确保流程执行规范、有序，避免出现随意跳转、遗漏环节或状态混乱的情况。

### 2.3.3 系统间职责明确分离
各系统按照职责边界承担相应功能。PS Admin 负责工单管理、流程驱动、状态控制及业务数据整合；外部系统负责支付、库存、物流等具体业务执行。各系统之间通过接口进行数据交互，确保职责清晰、协同有序。


## 2.4 多语言要求

系统需支持中文、英文、韩文三种语言切换。

# 3. 售后整体流程

## 3.1 主流程总览

本系统用于管理售后服务从申请到完成的全过程，主流程如下：

**售后申请 → 创建工单 → 接收 → 检测 → 判定 → 处理 → 返还客户 → 服务完成 → 满意度调查**

当检测后存在需客户确认或补充说明的情况时，可发起咨询；咨询完成后，继续进入判定流程。  
根据判定结果，工单可进入支付、维修、交换或结束等不同处理路径。

```mermaid
flowchart TD

A[售后申请] --> B[创建工单]

B --> C[接收]
C --> D[检测]
D --> E{判定结果}

C -. 可发起 .-> X[发起咨询]
X --> Y[客户确认/回复]
Y -. 记录结果 .-> E

E -->|有偿类| F[支付]
E -->|无偿类| G[进入维修 / 合作厂家处理]
E -->|交换类| H[进入交换流程]
E -->|无法维修/假货| I[判定结束]

F --> J{是否支付成功}
J -->|是| G
J -->|否| K[未支付结束]

G --> L[处理完成]
H --> L

L --> M[返还客户]
M --> N[服务完成]
N --> O[满意度调查]
```


## 3.2 流程说明

客户可通过小程序或门店发起售后申请，系统统一创建工单，并按接收、检测、判定、处理、履约、完成的顺序推进。

根据判定结果，工单可进入维修、交换、合作厂家 / 3PL 处理，或直接结束（如无法修理、假货等）。其中，有偿场景需先完成支付，再进入后续处理。

处理完成后，系统进入返还与交付阶段。客户签收或完成取件后，工单进入服务完成，并可继续进入满意度调查、问卷或回访流程。

整体流程可分为以下阶段：

**1）申请**  
客户通过小程序或门店发起售后申请，系统生成工单。

**2）接收与检测**  
产品完成寄送或接收后，进入接收与检测环节。

**3）判定与处理**  
系统根据检测结果进入维修、交换、外部处理或结束流程；如为有偿场景，需先完成支付。

**4）返还与交付**  
处理完成后，通过快递返还或门店取件完成交付。

**5）完成**  
客户签收或取件完成后，工单进入服务完成，并可继续进入满意度调查、问卷或回访流程。

系统可在接收完成、判定结果、支付状态、交付完成等关键节点触发客户通知。通知方式支持自动触发与人工发送，具体以业务配置为准。


## 3.3 关键分支流程

### 3.3.1 申请流程

客户可通过以下两种方式发起售后申请：

- 小程序申请（客户自行提交）
- 门店申请（门店协助提交）

申请完成后，系统生成工单，并进入收件流程。  
在进入维修流程前，客户可撤销申请。

```mermaid
flowchart TD
```

### 3.3.2 产品收件流程

系统应支持维修中心在收件场景下，根据有限信息快速定位待收货工单。查询条件应支持手机号、手机号后 4 位、快递单号、客户姓名、地址等信息，并支持模糊查询，以提升高频收件场景下的操作效率。

当查询结果存在多个候选工单时，系统应支持操作人员人工选择目标工单，并完成包裹与工单的绑定。绑定完成后，系统应自动生成内部条形码，并将工单状态更新为“已接收”。

客户在小程序提交售后申请后，寄送产品时需补充填写寄件快递单号。该物流单号应由小程序同步回传至 PS Admin，用于后续收件识别、工单匹配及物流跟踪。门店寄送场景亦应遵循相同逻辑，由门店录入或补充物流单号，并同步至 PS Admin。

考虑到维修中心日收件量较大（约每日 50～60 单），系统应支持内部条形码的批量打印能力，以满足集中收件后的统一打印和张贴需求。

目前现场尚未确定专用条形码打印设备，相关打印设备需后续确认或采购；同时，条形码打印格式、标签尺寸、打印样式等内容也需在后续阶段进一步确认。

### 3.3.4 咨询流程

以下场景可发起咨询：

- 维修方案需客户确认
- 费用变更
- 他品交换
- 异常情况说明
- 镜片补偿
- 合作厂家维修延迟通知

咨询/客户确认为流程旁路，不作为主状态流转的阻塞条件，工单推进权仍由 PS Admin 掌握。
如有异议电话沟通

### 3.3.5 支付流程

当工单判定为有偿场景时，系统进入支付流程。

支付流程如下：

1. 系统从 SAP 查询维修费用；
2. 系统生成待支付信息，并同步至支付侧；
3. 客户通过小程序支付；
4. 支付结果回传 PS Admin，用于更新工单状态。

系统根据支付状态触发对应通知及提醒机制。  
待支付状态下：

- 判定完成后 +D1 起，每天提醒 1 次，共 5 次；
- 第 29 天发送最终未付款提醒；
- 超时未支付则关闭工单。

取消规则如下：

1. 在进入“维修进行中”前，客户可以通过小程序取消申请；
2. 如存在未完成或需撤销的支付单，取消时需同步执行支付取消；
3. 可取消范围包括待支付、支付处理中、待维修等阶段；
4. 系统需记录取消时间、取消来源、取消原因及处理结果；
5. 如取消成功，工单进入取消关闭状态；如支付取消失败，则需人工介入处理。

说明：

- 支付流程是有偿场景下的独立子流程；
- 支付成功后，工单先进入待维修，再按规则进入维修中；
- 未支付与拒绝报价对应不同关闭路径。
- PS Admin 只根据**支付成功/失败结果**推进流程

### 3.3.6 维修执行流程
维修流程包括进入待维修、按时间规则进入维修中，以及维修处理完成后的结果提交。

进入维修中的时间规则如下：

- 无偿：判定完成 +D3 后进入维修中；
- 有偿：付款完成 +D3 后进入维修中；
- 有偿_3PL：付款完成 +D1 后进入维修中。

维修过程中如涉及配件更换，系统需执行配件选择、库存校验及配件消耗记录
在维修过程中，可能因为配件不足，无法继续维修，系统**手动修改**状态：**等待配件**，该状态不仅可能发生在判定后，也可能发生在维修过程中任意阶段。

默认时效可以有标准值（如 7~10 天），但实际过程中经常变化，因此“预计完成时间 / 预计寄出时间”应支持**手工修改**，系统**保留操作记录**。

配件库存统一在“维修完成提交”时扣减，以保证库存数据一致。

### 3.3.7 出库与履约流程

处理完成后，工单进入交付阶段。系统在出库准备、发货、配送、签收等关键节点同步更新状态，并向客户发送履约通知。

流程说明如下：

**1）出库准备**  
处理完成后，工单进入出库准备阶段，系统同步更新相关状态与数据。

**2）发货批次**  
当工单状态为“维修完成”时，系统于每日下午 3 点启动发货批次任务。未付款或已取消工单不参与批次处理。

**3）物流处理**  
批次任务执行后，系统生成快递申请、打印运单，并将运单号及物流信息回传至 PS Admin。

**4）手动录入物流信息**  
当订单达到“出库准备完成”后，PS Admin 可在当日下午 3 点至次日下午 3 点期间手动录入快递单号及物流信息，并同步至小程序。

**5）自动申请控制**  
如已在规定时间内手动录入物流信息，则次日下午 3 点不再自动生成快递申请。

**6）履约完成**  
快递场景下，配送完成后按工作日 +D1 进入“服务完成”；门店场景下，门店收货完成后按工作日 +D1 进入“服务完成”。

说明：
- 工单进入“服务完成”后，系统自动生成问卷并设置截止日期；
- 问卷由小程序展示，客户可在截止日期前填写；
- 提交后系统记录评分及反馈，并更新问卷状态；
- 超期未填写则问卷自动过期并关闭回访；
- 问卷与回访数据用于服务质量分析，不影响工单主状态。

```mermaid
F --> G[通知WMS仓储系统发货]
G --> H[出库信息回传至PS Admin]

E -->|其他物流方式| I[安排物流发货]
I --> J[获取物流单号]
J --> K[回传物流信息至PS Admin]

H --> L[同步物流信息至小程序]
K --> L

L --> M[进入物流配送]

M --> N{交付目的地}

N -->|客户签收| O[配送完成]
O --> P[等待工作日 +D1]
P --> Q[服务完成]

N -->|门店收货| R[门店收货完成]
R --> S[确认客户已取件]
S --> T[等待工作日 +D1]
T --> Q
```

#### 3.3.7.1 3PL 运单号返回说明
目前3PL 能不能回传物流单号还需要确认，可能有以下3种情况：

|情况|运单号来源|PS Admin 获取方式|
|---|---|---|---|
|情况一：3PL直接回传|3PL/WMS/TMS|直接回传给 PS Admin|
|情况二：先回传 SAP|3PL 先给 SAP|PS Admin 再从 SAP 获取|
|情况三：不返回真实运单号|仅返回发货状态或内部单号|PS Admin 只能记录状态|


### 3.3.8 问卷流程

服务完成后，系统生成满意度问卷，并由小程序向客户展示与收集反馈。  
PS Admin 负责接收问卷结果并更新问卷及回访信息。

### 3.3.9 流程的异常处理

系统需支持关键异常处理：

|异常类型|处理方式|
|---|---|
|支付失败|重新发起或关闭|
|支付超时|自动关闭工单|
|库存不足|阻断流程或触发申请|
|物流异常|人工介入处理|
|数据异常|系统校验与修复|
|咨询超时|取消关闭或人工跟进|
|维修取消|客户线上取消或管理员人工取消|

补充规则如下：

1. 在“维修进行中”至“维修完成”阶段，不支持客户自行取消；
2. 该阶段如需取消，由管理员手动执行；
3. 系统需记录取消原因、取消时间、操作人及处理结果；
4. 如已进入配送准备或发货批次，需先确认取消后的物流处理规则；
5. 如取消后仍需返还产品，应明确后续履约方式及状态流转规则。

说明：

- 异常处理覆盖主流程各关键节点；
- 异常是否可恢复，取决于具体业务规则；
- 无法恢复时，必须进入明确关闭状态，不允许流程悬空。

## 4.2 SAP 协同

### 4.2.1 产品主数据同步

为保证售后工单中的产品资料与 SAP 保持一致，PS Admin 需接收 SAP 提供的产品主数据，并更新本地产品档案。

此处“产品”包含售后维修中涉及的主商品、替换件、零部件等业务对象，不再单独区分“配件主数据”。

该接口主要用于同步：

* 产品ID
* 产品名称
* 型号
* 类别
* 颜色
* 上市日期
* 生产工厂
* 库存地点
* 状态

```mermaid
sequenceDiagram
    participant SAP as SAP
    participant PS as PS Admin

    SAP-->>PS: syncProductMaster(产品主数据)
    PS->>PS: 更新产品档案
```


### 4.2.2 产品库存查询

在维修准备、维修执行、产品更换或库存判断前，PS Admin 需向 SAP 查询产品库存，以判断库存是否充足。所有库存相关判断均以 SAP 返回结果为准。

此处“产品库存”统一覆盖：

* 主商品库存
* 更换件库存
* 零部件库存
* 3PL 场景库存

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP

    PS->>SAP: queryProductInventory(产品ID, 仓库, 库存地点)
    SAP-->>PS: 返回库存数量/可用库存/状态
    PS->>PS: 判断库存是否充足
```


### 4.2.3 3PL 库存查询

在“有偿_3PL”场景下，支付完成后，PS Admin 需向 SAP 查询 3PL 库存是否充足，再决定是否进入 3PL 发货流程。

如库存不足，则工单进入等待补货或人工协调状态。

由于库存统一按产品口径管理，因此这里查询的是 3PL 产品库存，不再区分配件库存。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP

    PS->>SAP: query3plInventory(产品ID, 3PL仓)
    SAP-->>PS: 返回3PL库存数量/可用状态
    PS->>PS: 判断是否可进入3PL发货
```


### 4.2.4 产品价格查询与维修费用生成

当工单判定为有偿维修时，PS Admin 需先向 SAP 获取产品价格基础数据，再按预设百分比规则自动计算最终维修费用，并生成报价结果用于后续支付流程。

在该场景下：

* SAP 提供产品价格基础值
* PS Admin 按规则自动计算最终维修费用

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP

    PS->>SAP: queryProductBasePrice(产品ID/型号/维修类型)
    SAP-->>PS: 返回产品价格基础值
    PS->>PS: 按预设百分比计算最终维修费用
```


### 4.2.5 维修完成后的产品消耗同步

在维修完成提交后，PS Admin 需将本次维修实际消耗的产品信息同步给 SAP，由 SAP 统一完成库存扣减。

此处“产品消耗”统一覆盖：

* 维修更换件
* 产品替换消耗

```mermaid
sequenceDiagram
    participant R as 维修人员
    participant PS as PS Admin
    participant SAP as SAP

    R->>PS: 提交维修完成
    PS->>PS: 汇总实际产品消耗
    PS->>SAP: postProductConsumption(工单号, 产品明细, 数量)
    SAP-->>PS: 返回扣减结果
    PS->>PS: 更新扣减状态
```


### 4.2.6 产品消耗冲销

如维修完成后发生撤销、异常回退或取消处理，已提交给 SAP 的产品消耗需支持冲销。PS Admin 发起冲销请求后，SAP 返回冲销结果，并同步更新工单状态。

主要用于异常恢复场景。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP

    PS->>SAP: reverseProductConsumption(原消耗记录ID, 冲销原因)
    SAP-->>PS: 返回冲销结果
    PS->>PS: 更新工单与库存处理状态
```


### 4.2.7 库存不足后的产品申请与到货回传

当维修过程中库存不足时，工单可能进入等待库存流程。此时，PS Admin 需要发起产品申请，或接收外部系统回传的申请状态与到货结果，以便在产品到位后恢复维修流程。

本节所述“产品申请”统一覆盖更换件申请、零部件申请及替换用产品申请，整体用于支撑产品申请创建、申请状态跟踪、产品到货 / 入库回传及等待库存后的流程恢复。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP

    PS->>SAP: createProductRequest(工单号, 产品, 数量)
    SAP-->>PS: 返回申请结果
    SAP-->>PS: productRequestStatusCallback(申请状态变化)
    SAP-->>PS: inboundResultCallback(已到货/已入库)
    PS->>PS: 更新等待库存状态并恢复维修流程
```


### 4.2.8 DO 创建

当工单满足 3PL 发货条件后，PS Admin 需调用 SAP 创建 DO 单据，由 SAP 向 WMS / 3PL 下发发货指令。PS Admin 记录 DO 编号及创建结果，用于后续履约推进。

该接口是 3PL 发货场景中的核心接口之一。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP
    participant WMS as WMS/3PL

    PS->>SAP: createDeliveryOrder(工单号, 收货信息, 产品明细)
    SAP->>SAP: 生成DO单据
    SAP->>WMS: 下发DO发货指令
    SAP-->>PS: 返回DO单号/创建结果
```


### 4.2.9 DO 状态查询

若 SAP 未主动回传 DO 当前状态，或回调失败导致状态缺失，PS Admin 可通过查询接口获取 DO 状态，并更新工单履约信息。

该接口建议作为回调机制的补偿查询能力保留。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP

    PS->>SAP: queryDeliveryOrderStatus(DO单号)
    SAP-->>PS: 返回DO状态
    PS->>PS: 更新工单履约状态
```


### 4.2.10 出库结果回传

WMS / 3PL 发货后，需将出库结果回传给 SAP，再由 SAP 或相关系统同步给 PS Admin。PS Admin 根据出库结果更新工单发货状态，并为后续物流展示和履约跟踪提供基础数据。

```mermaid
sequenceDiagram
    participant WMS as WMS/3PL
    participant SAP as SAP
    participant PS as PS Admin

    WMS-->>SAP: 返回出库结果
    SAP-->>PS: outboundResultCallback(出库状态, 时间, 明细)
    PS->>PS: 更新发货状态
```

### 4.2.11 3PL 运单号返回
目前3PL 能不能回传物流单号还需要确认，可能有以下3种情况：

#### 4.2.11.1 3PL 直接回传

在该场景下，3PL / WMS 在发货完成后，可直接向 PS Admin 返回承运商、真实运单号、发货时间等物流信息。PS Admin 再将物流信息同步至小程序，供客户查看真实物流轨迹。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP
    participant WMS as WMS/3PL

    PS->>SAP: createDeliveryOrder
    SAP->>WMS: 下发发货指令
    WMS-->>PS: 返回承运商/真实运单号/发货时间
```

#### 4.2.11.2 先回 SAP，再给 PS Admin

在该场景下，3PL / WMS 不直接向 PS Admin 返回物流信息，而是先回传给 SAP，再由 SAP 提供给 PS Admin。PS Admin 接收后同步至小程序。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP
    participant WMS as WMS/3PL

    PS->>SAP: createDeliveryOrder
    SAP->>WMS: 下发发货指令
    WMS-->>SAP: 返回承运商/真实运单号/发货时间
    SAP-->>PS: 返回物流信息
```

#### 4.2.11.3 仅返回发货状态

在该场景下，3PL / WMS 只能返回已发货状态或内部单号，无法返回客户可查询的真实快递运单号。PS Admin 只能先同步“已发货”状态，后续如有需要，再由人工补录真实运单号。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP
    participant WMS as WMS/3PL

    PS->>SAP: createDeliveryOrder(...)
    SAP->>WMS: 下发发货指令
    WMS-->>SAP: 返回已发货/内部单号
    SAP-->>PS: 返回发货状态
    PS->>PS: 手动补录真实运单号
```


### 4.2.12 交换出库

在产品交换或他品交换场景下，如需发出新产品，PS Admin 需调用 SAP 发起交换类出库处理。SAP 返回出库单据结果后，PS Admin 更新交换履约状态。

### 4.2.14 服务完成后结算单据回传

当工单取消或履约终止时，如前序已创建 DO 单据但尚未完成发货，PS Admin 需调用 SAP 执行 DO 取消，并根据返回结果更新工单状态。

## 4.3 物流系统协同

物流系统作为外部协同系统，主要负责售后过程中的寄件、运输、派送、签收及物流状态反馈，为 PS Admin 提供物流过程数据支持。


### 4.3.1 寄件与运输跟踪

客户或门店寄出产品后，物流系统负责运输，并将物流状态返回给 PS Admin。

```mermaid
sequenceDiagram
participant U as 客户 / 门店
```

## 4.4 通知协同

小程序通知适合用于工单状态更新、支付提醒、维修进度、发货进度及问卷邀请等轻量通知场景。  
但小程序订阅消息通常需由用户主动授权，常见的一次性订阅模式下，授权一次通常仅支持发送一条消息，因此不适合单独承担所有高频提醒场景。对于受理成功、支付催缴、无法维修、发货、到店提醒等关键节点，建议结合短信或邮件进行补充，以保证通知触达效果。

### 4.4.1 各通知渠道定位

| 通知渠道   | 适用场景                                         | 特点                             |
| ---------- | ------------------------------------------------ | -------------------------------- |
| 小程序通知 | 状态更新、支付提醒、维修进度、发货进度、问卷邀请 | 适合轻量提醒，可直接跳转业务页面 |
| 短信通知   | 受理成功、支付催缴、无法维修、发货、到店提醒     | 触达直接，适合关键节点通知       |
| 邮件通知   | 判定结果、处理方案、保修确认、出库说明、门店指引 | 信息完整，适合正式说明与留档     |

### 4.4.2 综合通知策略

| 业务节点                | 小程序通知 | 短信通知 | 邮件通知 | 推荐方式      | 说明                                     |
| ----------------------- | ---------- | -------- | -------- | ------------- | ---------------------------------------- |
| 工单受理完成            | 可发送     | 建议发送 | 可发送   | 短信 + 小程序 | 作为首次确认通知，需确保客户及时知晓     |
| 判定结果通知            | 可发送     | 可补充   | 建议发送 | 邮件 + 小程序 | 判定结果内容较多，邮件更适合承载详细说明 |
| 有偿维修待支付          | 建议发送   | 建议发送 | 可发送   | 小程序 + 短信 | 小程序承接支付操作，短信用于增强触达     |
| 支付催缴提醒            | 建议发送   | 建议发送 | 可选     | 短信 + 小程序 | 催缴场景建议双通道提醒                   |
| 支付完成                | 可发送     | 可发送   | 可发送   | 小程序 + 短信 | 用于通知客户已完成付款并进入后续流程     |
| 维修进行中              | 建议发送   | 可选     | 可选     | 小程序        | 适合轻量过程通知                         |
| 合作商维修延期          | 可发送     | 可发送   | 建议发送 | 邮件 + 短信   | 延期说明通常需要较完整内容               |
| 无法维修 / 假货         | 可发送     | 建议发送 | 建议发送 | 邮件 + 短信   | 属于关键结果通知，建议正式说明并强提醒   |
| 维修取消                | 可发送     | 建议发送 | 建议发送 | 邮件 + 短信   | 需明确取消原因及后续处理方式             |
| 更换产品 / 更换其他产品 | 可发送     | 可补充   | 建议发送 | 邮件 + 小程序 | 涉及方案说明，邮件更适合详细表达         |
| 保修卡 / 质保确认       | 可发送     | 可选     | 建议发送 | 邮件          | 适合承载说明与补充材料要求               |
| 出库发货                | 建议发送   | 建议发送 | 可发送   | 小程序 + 短信 | 发货状态建议及时触达，便于客户跟踪       |
| 发往门店                | 可发送     | 建议发送 | 可发送   | 短信 + 小程序 | 到店类通知建议短信补充                   |
| 门店到货 / 门店可取     | 可发送     | 建议发送 | 可选     | 短信 + 小程序 | 属于强提醒节点                           |
| 门店取货完成            | 可发送     | 可选     | 可选     | 小程序        | 用于结果确认即可                         |
| 问卷邀请                | 建议发送   | 可选     | 可选     | 小程序        | 最适合在小程序内直接承接填写动作         |
| 门店信息指引            | 不建议     | 可选     | 建议发送 | 邮件          | 内容较长，适合邮件承载                   |
