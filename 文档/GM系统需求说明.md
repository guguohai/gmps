# 1. 概述

## 1.1 项目背景

随着 Gentle Monster 中国区售后业务发展，售后流程已覆盖维修、配件、咨询、支付、物流、回访等多个环节，并涉及小程序、SAP、物流、短信等系统协同。随着业务链路和参与对象增加，现有管理方式对流程统一、状态透明、系统协同和过程追踪提出了更高要求。

因此，需要建设一套覆盖售后全流程的管理系统，统一承接售后工单、维修处理、配件库存协同及客户服务相关业务，打通相关系统数据，形成清晰、可控、可追踪的业务闭环。

## 1.2 项目目标

本项目旨在构建一套覆盖售后全流程的管理系统，作为客户服务的核心平台，支撑售后业务规范、高效运行。

具体目标如下：

1. 统一售后流程，规范申请、检测、判定、维修、支付、出库、履约、回访等业务环节。  
2. 建立统一状态管理机制，提高流程透明度和业务可控性。  
3. 打通与小程序、SAP、物流等系统的数据协同，提升处理效率。  
4. 支撑维修、配件、库存、咨询、问卷等业务的统一管理。  
5. 实现关键过程留痕，便于查询、追踪和问题定位。  


# 2. 业务范围与系统边界

## 2.1 业务范围

系统覆盖售后服务完整流程：

申请 → 受理 → 收货 → 检测 → 判定 → 处理 → 出库 → 履约 → 完成

支持以下业务场景：

- 维修（有偿 / 无偿）
- 产品更换（同品 / 他品）
- 合作厂家 / 3PL 处理
- 无法修理 / 假货处理
- 异常处理（支付 / 库存 / 物流等）
- 服务完成后的问卷与回访



## 2.2 系统边界

| 业务环节 | 责任系统       | 说明      |
| ---- | ---------- | ------- |
| 申请   | 小程序 / 门店   | 用户或门店发起 |
| 流程控制 | PS Admin   | 核心系统    |
| 支付   | 小程序 + 支付系统 | 完成支付    |
| 库存   | SAP        | 库存与出入库  |
| 物流   | 物流系统       | 配送与签收   |

说明：

- PS Admin 负责流程驱动与状态管理
- 外部系统负责具体业务执行
- 各系统通过接口进行数据协同


## 2.3 系统设计原则

### 2.3.1 所有业务以工单为核心
系统中的申请、受理、维修、支付、出库、完成及异常处理等业务，均以工单作为统一管理载体。工单作为业务处理主线，承载流程状态、业务数据及处理结果，确保全过程信息统一归集。

### 2.3.2 流程由状态驱动推进
系统流程通过状态进行驱动和控制，原则上按既定流转顺序逐步推进。各环节需在满足对应条件后进入下一状态，确保流程执行规范、有序，避免出现随意跳转、遗漏环节或状态混乱的情况。

### 2.3.3 系统间职责明确分离
各系统按照职责边界承担相应功能。PS Admin 负责工单管理、流程驱动、状态控制及业务数据整合；外部系统负责支付、库存、物流等具体业务执行。各系统之间通过接口进行数据交互，确保职责清晰、协同有序。


## 2.4 多语言要求

系统需支持中文、英文、韩文三种语言切换。

系统主要业务角色说明系统使用范围如下：

# 3. 售后整体流程

## 3.1 主流程总览

本系统用于管理售后服务从申请到完成的全过程，主流程如下：

**售后申请 → 创建工单 → 接收 → 检测 → 判定 → 处理 → 返还客户 → 服务完成 → 满意度调查**

当检测后存在需客户确认或补充说明的情况时，可发起咨询；咨询完成后，继续进入判定流程。  
根据判定结果，工单可进入支付、维修、交换或结束等不同处理路径。

```mermaid
flowchart TD

A[受理完成] --> B[接收]
B --> C[产品入库/判定]

C -.-> N[咨询等待]
N --> D{处理类型}

C --> D{处理类型}
D -->|有偿类| E[支付]
D -->|无偿类| F[进入处理]
D -->|交换类| G[进入交换流程]

E --> F
F --> J[处理完成]
G --> J

J --> K[返还客户]
K --> L[服务结束]
L -.-> M[满意度调查]

D -->|无法维修/假货| H[结束]
```


## 3.2 流程说明

客户可通过小程序或门店发起售后申请，系统统一创建工单，并按接收、检测、判定、处理、履约、完成的顺序推进。

根据判定结果，工单可进入维修、交换、合作厂家 / 3PL 处理，或直接结束（如无法修理、假货等）。其中，有偿场景需先完成支付，再进入后续处理。

处理完成后，系统进入返还与交付阶段。客户签收或完成取件后，工单进入服务完成，并可继续进入满意度调查、问卷或回访流程。

整体流程可分为以下阶段：

**1）申请**  
客户通过小程序或门店发起售后申请，系统生成工单。

**2）接收与检测**  
产品完成寄送或接收后，进入接收与检测环节。

**3）判定与处理**  
系统根据检测结果进入维修、交换、外部处理或结束流程；如为有偿场景，需先完成支付。

**4）返还与交付**  
处理完成后，通过快递返还或门店取件完成交付。

**5）完成**  
客户签收或取件完成后，工单进入服务完成，并可继续进入满意度调查、问卷或回访流程。

系统可在接收完成、判定结果、支付状态、交付完成等关键节点触发客户通知。通知方式支持自动触发与人工发送，具体以业务配置为准。


## 3.3 关键分支流程

### 3.3.1 申请流程

客户可通过以下两种方式发起售后申请：

- 小程序申请（客户自行提交）
- 门店申请（门店协助提交）

申请完成后，系统生成工单，并进入收件流程。  
在进入维修流程前，客户可撤销申请。

```mermaid
flowchart TD

A[客户发起售后申请] --> B{申请入口}

B -->|微信小程序| C[客户在小程序提交申请]
B -->|门店接待| D{是否可使用小程序}

D -->|可以| C
D -->|不可以| E[门店联系PS Admin]
E --> F[PS Admin代客户录入申请]

C --> G[进入微信审核]

H -->|通过| I[创建售后订单]
H -->|不通过| J[记录驳回原因并传回PS Admin]

F --> I
```


### 3.3.2 产品收件流程

客户或门店发起售后申请并生成工单后，进入产品收件阶段。客户寄件时，需补充填写寄件快递公司及运单号；门店寄送场景下，也应由门店录入或补充寄件物流信息，并同步至 PS Admin，用于后续收件识别、工单匹配及物流跟踪。

维修中心在收件时，系统应支持根据手机号、手机号后 4 位、快递单号、客户姓名、地址等信息快速查询待收货工单，并支持模糊检索，以提升高频收件场景下的操作效率。

当查询结果存在多个候选工单时，系统应支持操作人员人工确认目标工单，并完成包裹与工单绑定。绑定完成后，系统自动生成内部条形码、自动分配维修单号，并将工单状态更新为“已收到受理”。

条形码标签支持首次自动打印；如因标签损坏、丢失或补打需要，系统也应支持操作人员后续手动打印。条形码编码规则应包含以下信息：

- 日期
- 当天顺序编号
- 客户手机号后四位

条形码除用于标识包裹外，还应用于后续操作场景。系统应支持通过扫描条形码直接查询工单、跳转工单详情，并在指定录入场景中支持扫码后直接保存数据，以提升现场操作效率。

### 3.3.3 检测与判定流程

产品检测完成后进入判定阶段，系统根据检测结果确定后续处理方式，包括：
- 无偿维修
- 有偿维修
- 产品交换
- 合作厂家处理
- 有偿_3PL
- 无法修理
- 假货处理
- 他品交换
系统根据判定结果向客户发送对应通知，告知处理方案及后续安排。

在“有偿_3PL”场景下，支付完成后仍需先进行 3PL 库存判断；若库存不足，则不可直接进入 3PL 发货流程，需进入等待补货或人工协调状态。

### 3.3.4 咨询流程

以下场景可发起咨询：

- 维修方案需客户确认
- 费用变更
- 他品交换
- 异常情况说明
- 镜片补偿
- 合作厂家维修延迟通知
- 产品状态确认
- 其他需客户补充确认的信息

咨询/客户确认为流程旁路，不作为主流程阻塞节点，但咨询结果将作为后续判定与处理的重要依据。

系统发起咨询时，应自动记录咨询日期、咨询负责人及咨询状态。咨询内容可包括文字说明、图片信息及客户确认项。对于需客户查看产品状态并确认的场景，系统应支持将图片及文字说明同步至小程序，由客户完成确认或反馈后回写至 PS Admin。

如咨询完成，工单返回后续判定或处理节点继续推进；如存在电话沟通或其他线下沟通方式，系统也应保留对应处理记录。

```mermaid
flowchart TD

A[发起咨询] --> B[同步咨询内容至小程序/电话/线下沟通]
```

### 3.3.5 支付流程

当工单判定为有偿场景时，系统进入支付流程。

支付流程如下：

1. 系统从 SAP 获取价格基础数据；
2. PS Admin 按业务规则生成维修费用与待支付信息；
3. 待支付信息同步至支付侧；
4. 客户通过小程序支付；
5. 支付结果回传 PS Admin，用于更新工单状态。

系统根据支付状态触发对应通知及提醒机制。未支付场景下，系统需支持催缴提醒，并在超时未支付时关闭工单。

取消规则如下：

1. 在进入“维修进行中”前，客户可以通过小程序取消申请；
2. 如存在未完成或需撤销的支付单，取消时需同步执行支付取消；
3. 可取消范围包括待支付、支付处理中、待维修等阶段；
4. 系统需记录取消时间、取消来源、取消原因及处理结果；
5. 如取消成功，工单进入取消关闭状态；如支付取消失败，则需人工介入处理。

状态控制规则如下：

- 在待支付状态下，不允许工单直接跳转至维修进行中、维修完成、出库完成、配送完成等后续状态；
- 仅当支付成功后，工单方可进入后续维修或处理链路；
- 未支付关闭与客户拒绝报价应进入不同关闭路径，并分别记录原因。

说明：

- 支付流程是有偿场景下的独立子流程；
- 支付成功后，工单先进入待维修，再按规则进入维修中；
- PS Admin 仅根据支付最终结果推进后续流程；
- 具体状态推进时点、提醒频次及状态控制规则，统一参见 3.4《状态自动化规则》。


```mermaid
flowchart TD

A[判定为有偿 → 查询维修费用] --> B[生成支付订单]
B --> C[微信小程序向用户推送支付信息]

C --> E{用户是否接受报价}

E -->|接受| F[进入待支付]
E -->|拒绝| P[关闭订单]

F --> G{是否完成支付}

```


### 3.3.6 维修执行流程
维修流程包括进入待维修、按规则进入维修中，以及维修处理完成后的结果提交。

系统应支持根据不同处理类型进入对应维修链路。无偿、有偿与有偿_3PL 场景在进入维修中的条件上存在差异，具体自动推进规则统一参见 3.4《状态自动化规则》。

系统应支持在进入维修处理后自动填写维修开始日期，并记录维修负责人。对于预计完成时间、预计寄出时间等时效字段，系统可根据默认规则自动生成初始值，但应支持人工修改，并保留修改记录。

维修过程中如涉及配件更换，系统需执行配件选择、库存校验及配件消耗记录。  
如配件库存不足，工单可进入“等待配件”状态，待配件到位后恢复维修流程。  
配件与产品消耗不在预检判定阶段扣减，而在维修完成提交时，根据本次实际维修内容统一生成消耗记录并执行库存扣减。

当维修详情为“产品更换”时，系统应根据维修内容自动生成对应消耗记录；当维修详情为“更换其他产品”时，系统应自动生成新产品及相关配件的消耗记录。一个维修单可对应多个实际消耗项。

```mermaid
flowchart TD

A[维修中] --> B[准备配件]

B --> G[库存校验]
G --> H{库存是否充足}

H -->|是| E[维修执行]
E --> I[维修完成提交]
I --> T[待出库]
T --> U[确认已发货]
U --> J[PS Admin提交配件消耗]
J --> K[SAP统一扣减配件库存]

H -->|否| N{处理方式}
N -->|配件申请| O[触发配件申请流程]
O --> P[等待配件到位]
P --> G

N -->|库存预警| Q[生成库存预警]
Q --> R[人工处理后继续]
R --> G
```


### 3.3.7 出库与履约流程

处理完成后，工单进入交付阶段。系统在出库准备、发货、配送、签收等关键节点同步更新状态，并向客户发送履约通知。

流程说明如下：

**1）出库准备**  
处理完成后，工单进入出库准备阶段，系统同步更新相关状态与数据。

**2）发货批次**  
当工单状态为“维修完成”时，系统于每日下午 3 点启动发货批次任务。未付款或已取消工单不参与批次处理。

**3）物流处理**  
批次任务执行后，系统生成快递申请、打印运单，并将运单号及物流信息回传至 PS Admin。

**4）手动录入物流信息**  
当订单达到“出库准备完成”后，PS Admin 可在当日下午 3 点至次日下午 3 点期间手动录入快递单号及物流信息，并同步至小程序。

**5）自动申请控制**  
如已在规定时间内手动录入物流信息，则次日下午 3 点不再自动生成快递申请。

**6）履约完成**  
快递场景下，配送完成后进入“服务完成”；门店场景下，门店收货完成后进入“服务完成”。

说明：

- 工单进入“服务完成”后，系统自动生成问卷并设置截止日期；
- 问卷由小程序展示，客户可在截止日期前填写；
- 提交后系统记录评分及反馈，并更新问卷状态；
- 超期未填写则问卷自动过期并关闭回访；
- 问卷与回访数据用于服务质量分析，不影响工单主状态；
- 具体批处理时点、状态推进时点及自动完成规则，统一参见 3.4《状态自动化规则》。


```mermaid
flowchart TD

A[出库准备完成] --> B{是否等待<br>发货批次任务}

B -->|否| C[PS Admin 手动录入发货物流信息]
B -->|是| D[下午3点发货批次任务]

C --> E{发货方式}
D --> E

E -->|3PL| F[SAP生成DO并通知WMS发货]
F --> G[出库信息回传至PS Admin]

E -->|其他物流方式| H[安排物流发货并获取物流单号]
H --> I[回传物流信息至PS Admin]

G --> J[同步物流信息至小程序]
I --> J

J --> K[进入物流配送]

K --> L{交付目的地}

L -->|客户签收| M[配送完成]

```

#### 3.3.7.1 3PL 运单号返回说明
目前3PL 能不能回传物流单号还需要确认，可能有以下3种情况：

|情况|运单号来源|PS Admin 获取方式|
|---|---|---|---|
|情况一：3PL直接回传|3PL/WMS/TMS|直接回传给 PS Admin|
|情况二：先回传 SAP|3PL 先给 SAP|PS Admin 再从 SAP 获取|
|情况三：不返回真实运单号|仅返回发货状态或内部单号|PS Admin 只能记录状态|


### 3.3.8 问卷流程

服务完成后，系统生成满意度问卷，并由小程序向客户展示与收集反馈。  
PS Admin 负责接收问卷结果并更新问卷及回访信息。

问卷生成后，系统应同时记录问卷发送时间、有效截止日期及问卷状态。客户在截止日期前提交问卷时，系统更新为“已回复”；超过截止日期仍未提交的，系统自动更新为“已过期”，并关闭回访流程。

### 3.3.9 流程的异常处理

系统需支持关键异常处理：

|异常类型|处理方式|
|---|---|
|支付失败|重新发起或关闭|
|支付超时|自动关闭工单|
|库存不足|阻断流程或触发申请|
|物流异常|人工介入处理|
|数据异常|系统校验与修复|
|咨询超时|取消关闭或人工跟进|
|维修取消|客户线上取消或管理员人工取消|

补充规则如下：

1. 在“维修进行中”至“维修完成”阶段，不支持客户自行取消；
2. 该阶段如需取消，由管理员手动执行；
3. 系统需记录取消原因、取消时间、操作人及处理结果；
4. 如已进入配送准备或发货批次，需先确认取消后的物流处理规则；
5. 如取消后仍需返还产品，应明确后续履约方式及状态流转规则；
6. 对于状态自动化执行失败、批处理失败或数据未按规则更新的场景，系统应保留异常记录并支持人工修正；
7. 无法恢复时，必须进入明确关闭状态，不允许流程悬空。

### 3.4.3 条码规则

系统应支持条码的生成、打印、识别及业务操作联动，具体规则如下：

1. 收件绑定完成后自动生成条码编号；
    
2. 条码编号规则应包含日期、当天顺序编号及客户手机号后四位；
    
3. 首次生成后自动打印条码标签；
    
4. 系统支持后续手动补打；
    
5. 支持通过扫描条码直接查询工单；
    
6. 支持通过扫描条码直接跳转工单详情；
    
7. 在指定录入场景中，支持录入完成后通过扫描条码直接保存数据。
    


### 3.4.4 状态控制规则

系统应对关键状态设置控制规则，避免流程越级推进或数据口径不一致，具体如下：

1. 付款待机状态下，不允许直接进入维修进行中、维修完成、出库完成、配送完成等后续状态；
    
2. 未完成支付的有偿工单，不允许直接进入维修链路；
    
3. 库存不足时，不允许直接进入 3PL 发货或维修完成；
    
4. 咨询未完成时，系统可保持咨询待机，但需记录当前处理状态；
    
5. 配件/产品消耗不得在预检判定阶段直接扣减，统一在维修完成提交时处理；
    
6. 自动化规则执行失败时，系统应记录异常日志并支持人工修正。
    


### 3.4.5 时间规则说明

系统中涉及的 +D1、+D3、固定时点批处理及节假日排除等规则，应统一由系统配置管理。涉及默认时效的业务规则如下：

- 无偿：判定完成 +D3 进入维修进行中；
    
- 有偿：付款完成 +D3 进入维修进行中；
    
- 有偿_3PL：付款完成 +D1 进入维修进行中或进入 3PL 处理判断；
    
- 维修完成：工作日下午 3 点进入出库准备完成；
    
- 出库准备完成：当日下午 6 点进入配送开始；
    
- 配送完成 / 门店收货完成：工作日 +D1 进入服务完成；
    
- 预计出库日期：默认按排除节假日后的天数计算。
    
关于“预计出库日期”的具体起算点、节假日口径及批处理执行时点，应在项目实施阶段结合客户业务规则统一确认并配置。

# 4.2 SAP 协同

## 4.2.1 协同说明

在本项目中，PS Admin 作为售后工单管理核心系统，负责售后工单创建、状态流转、业务处理控制及与小程序侧的协同展示；SAP 主要负责产品主数据、库存数据、价格基础数据、3PL 出库处理、交换出库、原品退回收货以及服务完成后的 SO 单据处理。

双方通过业务数据同步与结果回传，共同完成售后流程闭环。

本章节主要说明 PS Admin 与 SAP 在售后业务中的协同内容，重点面向业务评审，不展开具体技术接口定义。


## 4.2.2 产品主数据同步

为保证售后工单中的产品资料与 SAP 保持一致，PS Admin 需定时接收 SAP 提供的产品主数据，并更新本地产品档案。

同步内容主要包括：

* 产品 ID
* 产品名称
* 型号
* 类别
* 上市日期
* 生产工厂
* 库存地点
* 状态

该能力用于确保售后工单、产品识别、库存判断及后续处理过程中的产品信息统一，以 SAP 数据为准。

```mermaid
sequenceDiagram
    participant SAP as SAP
    participant PS as PS Admin

    SAP-->>PS: 定时同步产品主数据
    PS->>PS: 更新本地产品档案
```


## 4.2.3 产品库存同步与判断

在维修准备、维修执行、产品更换、他品交换或 3PL 发货等场景下，PS Admin 需基于 SAP 库存数据进行库存判断，以确认后续业务是否可以继续执行。

库存相关数据由 SAP 定时同步或定时提供查询结果，PS Admin 依据同步结果进行业务判断，所有库存口径以 SAP 为准。

此处“产品库存”统一覆盖：

* 主商品库存
* 更换件库存
* 零部件库存
* 3PL 场景库存

当 SAP 判断库存不足时，工单不可直接进入后续处理，需进入等待补货或人工协调状态，待库存满足后再恢复流程。

```mermaid
sequenceDiagram
    participant SAP as SAP
    participant PS as PS Admin

    SAP-->>PS: 定时同步库存数据 / 定时提供库存结果
    PS->>PS: 判断库存是否满足后续处理条件
```


## 4.2.4 3PL 场景库存判断

在“有偿_3PL”场景下，客户支付完成后，PS Admin 需进一步确认 3PL 库存是否充足，再决定是否进入 3PL 发货流程。

如库存充足，则进入后续 3PL 履约处理；如库存不足，则工单进入等待补货或人工协调状态。

该判断仍以 SAP 提供的库存数据为准。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP

    PS->>PS: 支付完成

    PS->>PS: 判断是否进入3PL发货流程
```

## 4.2.10 3PL 运单号返回

目前 3PL 是否能够回传真实物流运单号，未最终确认。结合现阶段情况，先按以下 3 种可能方式进行说明，最终以客户确认后的实际模式为准。

### 4.2.10.1 3PL 直接回传物流信息

在该场景下，3PL / WMS 在发货完成后，可直接返回承运商、真实运单号、发货时间等物流信息，PS Admin 接收后用于物流展示与履约跟踪。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP
    participant WMS as WMS/3PL

    PS-->>SAP: 提交3PL发货需求
    SAP->>WMS: 下发发货处理
    WMS-->>PS: 返回承运商/真实运单号/发货时间
```

### 4.2.10.2 3PL 先回 SAP，再同步给 PS Admin

在该场景下，3PL / WMS 不直接向 PS Admin 返回物流信息，而是先返回给 SAP，再由 SAP 同步给 PS Admin。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP
    participant WMS as WMS/3PL

    PS-->>SAP: 提交3PL发货需求
    SAP->>WMS: 下发发货处理
    WMS-->>SAP: 返回承运商/真实运单号/发货时间
    SAP-->>PS: 同步物流信息
```

### 4.2.10.3 仅返回发货状态，不返回真实运单号

在该场景下，3PL / WMS 仅返回已发货状态或内部单号，无法返回可查询的真实物流运单号。PS Admin 先同步“已发货”状态，后续如业务需要，再由人工补录真实运单号。

```mermaid
sequenceDiagram
    participant PS as PS Admin
    participant SAP as SAP
    participant WMS as WMS/3PL

    PS-->>SAP: 提交3PL发货需求
    SAP->>WMS: 下发发货处理
    WMS-->>SAP: 返回已发货状态/内部单号
    SAP-->>PS: 同步发货结果
    PS->>PS: 视需要人工补录真实运单号
```

## 4.4 通知协同

小程序通知适合用于工单状态更新、支付提醒、维修进度、发货进度及问卷邀请等轻量通知场景。  
但小程序订阅消息通常需由用户主动授权，常见的一次性订阅模式下，授权一次通常仅支持发送一条消息，因此不适合单独承担所有高频提醒场景。对于受理成功、支付催缴、无法维修、发货、到店提醒等关键节点，建议结合短信或邮件进行补充，以保证通知触达效果。

### 4.4.1 各通知渠道定位

| 通知渠道   | 适用场景                                         | 特点                             |
| ---------- | ------------------------------------------------ | -------------------------------- |
| 小程序通知 | 状态更新、支付提醒、维修进度、发货进度、问卷邀请 | 适合轻量提醒，可直接跳转业务页面 |
| 短信通知   | 受理成功、支付催缴、无法维修、发货、到店提醒     | 触达直接，适合关键节点通知       |
| 邮件通知   | 判定结果、处理方案、保修确认、出库说明、门店指引 | 信息完整，适合正式说明与留档     |

### 4.4.2 综合通知策略
|业务节点|推荐方式|说明|
|---|---|---|
|工单受理完成|短信 + 小程序|作为首次确认通知，需确保客户及时知晓|
|判定结果通知|邮件 + 小程序|判定结果内容较多，邮件更适合承载详细说明|
|有偿维修待支付|小程序 + 短信|小程序承接支付操作，短信用于增强触达|
|支付催缴提醒|短信 + 小程序|催缴场景建议双通道提醒|
|支付完成|小程序 + 短信|用于通知客户已完成付款并进入后续流程|
|维修进行中|小程序|适合轻量过程通知|
|合作商维修延期|邮件 + 短信|延期说明通常需要较完整内容|
|无法维修 / 假货|邮件 + 短信|属于关键结果通知，建议正式说明并强提醒|
|维修取消|邮件 + 短信|需明确取消原因及后续处理方式|
|更换产品 / 更换其他产品|邮件 + 小程序|涉及方案说明，邮件更适合详细表达|
|保修卡 / 质保确认|邮件|适合承载说明与补充材料要求|
|出库发货|小程序 + 短信|发货状态建议及时触达，便于客户跟踪|
|发往门店|短信 + 小程序|到店类通知建议短信补充|
|门店到货 / 门店可取|短信 + 小程序|属于强提醒节点|
|门店取货完成|小程序|用于结果确认即可|
|问卷邀请|小程序|最适合在小程序内直接承接填写动作|
|门店信息指引|邮件|内容较长，适合邮件承载|

#### （11）库存预警与提醒

**展示位置**
1. 在工单详情页顶部显示工单级库存预警提示。
2. 在“库存申请”按钮附近可显示简要异常标识，如“库存不足”“3PL库存预警”。
3. 在涉及处理方式选择、3PL发货选择、配件选择等操作区域内，应再次展示对应库存信息及预警提示。

**提示规则**
1. 当选择 3PL 发货路径时，系统应显示对应 3PL 库存数量。
2. 当库存低于预设阈值时，系统应显示库存不足提醒。
3. 当库存为 0 或不满足当前处理条件时，系统应阻止继续选择该处理路径，并提示需等待补货或改为其他处理方式。
4. 当库存恢复后，系统应支持刷新最新库存结果，并允许继续后续操作。

**提示内容示例**
- 当前 3PL 库存不足，请确认是否调整处理路径。
- 当前无可用库存，无法选择 3PL 发货。
- 当前库存不足，可发起库存申请或等待补货。
