# 售后系统需求说明（客户评审版）

---

# 1. 项目背景与目标

为支撑中国区售后业务发展，需要建设一套统一的售后管理系统，实现售后服务全流程的标准化管理与多系统协同。

系统建设目标：

- 建立统一的售后流程与状态管理体系
- 提升售后工单处理效率
- 打通小程序、SAP、物流及支付系统
- 提供清晰的服务进度与客户交互能力
- 支撑售后数据统计与运营分析

---

# 2. 业务范围与系统边界

## 2.1 业务范围

系统覆盖售后服务完整流程：

申请 → 受理 → 收货 → 检测 → 判定 → 处理 → 出库 → 履约 → 完成

支持以下业务场景：

- 维修（有偿 / 无偿）
- 产品更换（同品 / 他品）
- 合作厂家 / 3PL 处理
- 无法修理 / 假货处理
- 异常处理（支付 / 库存 / 物流等）
- 服务完成后的问卷与回访

---

## 2.2 系统边界

|业务环节|责任系统|说明|
|---|---|---|
|申请|小程序 / 门店|用户或门店发起|
|流程控制|PS Admin|核心系统|
|支付|小程序 + 支付系统|完成支付|
|库存|SAP|库存与出入库|
|物流|物流系统|配送与签收|

说明：

- PS Admin 负责流程驱动与状态管理
- 外部系统负责具体业务执行
- 各系统通过接口进行数据协同

---

## 3.2 流程说明

客户通过小程序或门店发起售后申请，PS Admin 生成工单并完成收件与检测。

根据检测结果，工单进入维修（有偿需先支付）、换货、3PL处理或直接结束。

处理完成后同步 SAP，并通过快递或门店完成交付。

客户签收后流程结束，系统发送满意度调查。

---

## 3.3 关键分支流程

### 3.3.1 申请流程

用户提交售后申请后生成工单，寄件并被维修中心签收入库后进入检测阶段。  
售后申请支持两种入口：

- 小程序申请
    
- 门店申请
    

```mermaid
flowchart TD

A1[用户在小程序发起申请] --> A2[提交售后信息]
A2 --> A3[生成工单（RECEIVED）]
A3 --> A4[用户填写寄件信息]
A4 --> A5[等待收货（WAIT_RECEIVE）]

B1[客户到门店申请] --> B2[门店创建工单]
B2 --> B3[生成工单（RECEIVED）]
B3 --> B4[门店代寄或客户寄件]
B4 --> B5[等待收货（WAIT_RECEIVE）]

A5 --> C1[维修中心签收]
B5 --> C1
C1 --> C2[入库完成（RECEIVED_WAREHOUSE）]
C2 --> C3[进入检测（INSPECTING）]
```

说明：

- 小程序与门店为两个申请入口，但统一由 PS Admin 生成工单
    
- 进入检测前必须完成签收与入库
    
- 申请流程结束后，工单进入检测主流程
    

### 3.3.2 产品接收流程

```mermaid
flowchart TD

A1[用户/门店寄出产品] --> A2[填写运单号]
A2 --> A3[运输中]
A3 --> A4[维修中心收货]
A4 --> A5[收货登记]
A5 --> A6[入库完成（RECEIVED_WAREHOUSE）]

B1[客户到门店交付产品] --> B2[门店验收]
B2 --> B3[门店登记收货]
B3 --> B4[门店寄往维修中心]
B4 --> A3

A4 --> C1{是否异常}
C1 -->|否| A5
C1 -->|是| C2[记录异常]
C2 --> C3[通知客户/门店]
C3 --> C4[异常处理完成]

A6 --> D1[进入检测（INSPECTING）]
```

说明：

- 快递寄送与门店直收两种收货方式最终汇合为维修中心收货
    
- 产品收货异常需单独记录，并通知相关方
    
- 收货完成后进入检测状态
    

### 3.3.3 检测与判定流程

检测完成后进入判定，根据不同结果流转到不同处理路径。

```mermaid
flowchart TD

A[检测中（INSPECTING）] --> B[检测完成 → 待判定（WAIT_DECISION）]
B --> C{判定结果}

C --> D[无偿维修（WAIT_REPAIR）]
C --> E[有偿维修 → 生成报价]
E --> F[等待支付（WAIT_PAYMENT）]

C --> G[产品更换处理中（EXCHANGE_PROCESSING）]
C --> H[3PL处理 → 进入维修流程（WAIT_REPAIR）]
C --> I[合作厂家维修 → 进入维修流程（WAIT_REPAIR）]

C --> J[无法修理（UNREPAIRABLE）]
C --> K[假货（FAKE_PRODUCT）]

C --> L[需咨询（WAIT_CUSTOMER_CONFIRM）]
```

说明：

- 检测完成后统一进入待判定状态
    
- 判定结果决定是否进入支付、维修、换货、关闭等不同路径
    
- 咨询为判定前后的可插入子流程
    

### 3.3.4 咨询流程

咨询状态默认值为“咨询待机”；当“咨询完成”后，状态自动返回判定节点。

在以下场景需发起咨询：

- 维修方案需客户确认
    
- 费用变更
    
- 他品交换
    
- 合作厂家延迟
    

```mermaid
flowchart TD

A[待客户确认（WAIT_CUSTOMER_CONFIRM）] --> B[发送咨询通知]
B --> C[客户确认/回复]
C --> D{是否确认}

D -->|是| E[返回判定（WAIT_DECISION）]
D -->|否/超时| F[取消工单（CLOSED_CANCELLED）]
```

说明：

- 咨询流程本质上是判定流程中的补充确认子流程
    
- 客户确认后返回待判定状态
    
- 若拒绝或超时，可进入取消关闭
    

### 3.3.5 支付流程

当工单判定为有偿维修场景时，系统进入支付流程。

```mermaid
flowchart TD

A[生成维修费用 WAIT_DECISION] --> B[生成支付订单 WAIT_PAYMENT]
B --> C[小程序查询支付订单 WAIT_PAYMENT]
C --> D[用户查看报价 WAIT_PAYMENT]

D --> E{是否接受报价}

E -->|接受| F[进入待支付 WAIT_PAYMENT]
F --> G[用户发起支付 WAIT_PAYMENT]
G --> H[微信支付完成 WAIT_PAYMENT]

H --> I[小程序接收回调并校验 WAIT_PAYMENT]
I --> J[通知PS Admin WAIT_PAYMENT]

J --> K[更新为已支付 PAID]
K --> L[进入维修流程 WAIT_REPAIR]

E -->|未支付| M[系统催缴提醒 WAIT_PAYMENT]
M --> N{30天内是否支付}

N -->|是| G
N -->|否| O[未付款退回关闭 CLOSED_UNPAID_RETURN]

E -->|拒绝| P[取消关闭 CLOSED_CANCELLED]
```

说明：

- 支付流程是有偿场景的独立子流程
    
- 支付成功后并不直接进入维修中，而是先进入待维修状态
    
- 未支付和拒绝报价对应不同关闭路径
    

### 3.3.6 维修执行流程

若为 “无偿”：判定完成 +D3 后状态自动变更为 “维修进行中”；
若为 “有偿”：付款完成 +D3 后状态自动变更为 “维修进行中”；
若为 “3PL”：付款完成 +D1 后状态自动变更为 “维修进行中”。

```mermaid
flowchart TD

A[判定完成（JUDGED）] --> B{维修类型是否有偿}

B -->|无偿| C[等待至 判定完成 +D3]
B -->|有偿| D{是否已付款（PAID）}
B -->|3PL| E{是否已付款（PAID）}

D -->|是| F[等待至 付款完成 +D3]
D -->|否| Z1[保持待维修（WAIT_REPAIR）]

E -->|是| G[等待至 付款完成 +D1]
E -->|否| Z1

C --> H[状态自动变更为 维修进行中（REPAIRING）]
F --> H
G --> H

H --> I[维修人员领取设备]
I --> J[填写维修开始日期]
J --> K[执行维修处理]

K --> L{维修处理类型}
L -->|维修处理| M[执行维修操作]
L -->|产品更换| N[扣减对应配件]
L -->|他品更换| O[扣减新产品库存]

M --> P[记录维修结果]
N --> P
O --> P

P --> Q[维修完成]
Q --> R[待出库（WAIT_OUTBOUND）]
```

说明：

- 待维修状态用于承接支付完成或无偿判定完成后的统一入口
    
- 满足时间规则后进入维修中
    
- 维修完成后进入待出库，而不是直接出库
    

### 3.3.7 配件与库存处理流程

在维修处理过程中，若涉及配件更换，系统需执行配件选择、库存校验及配件消耗记录等操作。  
该流程作为维修流程中的子流程执行。

1. 配件选择规则  
    维修人员可对系统默认配件进行确认、修改、新增或删除。
    
2. 库存校验规则  
    系统在维修完成前校验配件库存是否满足消耗需求。
    
3. 库存扣减时机  
    配件库存在“维修完成提交时”统一扣减，避免在维修过程中产生库存不一致问题。
    
4. 库存不足处理  
    当库存不足时，系统应支持：
    
    - 触发配件申请流程
        
    - 或生成库存预警
        
5. 数据记录  
    系统需记录：
    
    - 配件消耗明细
        
    - 库存变动记录
        
    - 操作人及时间
        

```mermaid
flowchart TD

A[维修中（REPAIRING）] --> B[选择配件]

B --> C[确认/修改/新增/删除配件]
C --> D[形成配件清单]

D --> E[维修执行]

E --> F[提交维修完成]
F --> G[库存校验]

G --> H{库存是否充足}

H -->|是| I[统一扣减库存]
I --> J[记录库存变动]
J --> K[记录配件消耗明细]
K --> L[记录操作人及时间]
L --> M[维修完成（WAIT_OUTBOUND）]

H -->|否| N{处理方式}

N -->|配件申请| O[触发配件申请流程]
O --> P[等待配件到位]
P --> G

N -->|库存预警| Q[生成库存预警]
Q --> R[人工处理后继续]
R --> G
```

说明：

- 配件与库存流程为维修执行子流程，不单独形成主状态链
    
- 库存扣减统一发生在维修完成提交时
    
- 库存不足时需阻断维修完成提交
    

### 3.3.8 合作厂家 / 3PL 流程

```mermaid
flowchart TD

A[待维修（WAIT_REPAIR）] --> B[发往合作厂家/3PL]
B --> C[维修中（REPAIRING）]
C --> D[厂家维修完成]
D --> E[返回仓库]
E --> F[待出库（WAIT_OUTBOUND）]
```

说明：

- 合作厂家 / 3PL 流程在业务上是外部执行，在系统状态上仍纳入维修主链
    
- 处理完成后统一返回待出库状态
    
- 若涉及费用，仍需先经过支付流程
    

### 3.3.9 无法修理 / 假货流程

```mermaid
flowchart TD

A[无法修理（UNREPAIRABLE）] --> B[通知客户]
B --> C[关闭（CLOSED_UNREPAIRABLE）]

D[假货（FAKE_PRODUCT）] --> E[通知客户]
E --> F[关闭（CLOSED_FAKE）]
```

说明：

- 无法修理与假货均为终止类处理路径
    
- 需先通知客户，再进入对应关闭状态
    
- 不再进入维修、出库和完成流程
    

### 3.3.10 出库与履约流程

```mermaid
flowchart TD

A[待出库（WAIT_OUTBOUND）] --> B[执行出库（OUTBOUND）]
B --> C{交付方式}

C -->|物流| D[运输中]
D --> E[客户签收（DELIVERED）]
E --> F[服务完成（COMPLETED）]

C -->|门店自提| G[门店到货（STORE_ARRIVED）]
G --> H[通知取件]
H --> I[客户取件（STORE_PICKED_UP）]
I --> F
```

说明：

- 出库后按履约方式区分快递与门店自提
    
- 快递签收与门店取件后均进入已完成
    
- 已完成代表服务业务流程结束
    

### 3.3.11 问卷流程

```mermaid
flowchart TD

A[已完成（COMPLETED）] --> B[系统生成问卷（SURVEY_PENDING）]
B --> C[发送问卷（SURVEY_SENT）]
C --> D[小程序展示问卷]

D --> E{是否在截止日期前填写}

E -->|是| F[提交问卷（SURVEY_REPLIED）]
F --> G[更新回访状态（FOLLOW_UP_COMPLETED）]
G --> H[完成关闭（CLOSED_COMPLETED）]

E -->|否| I[问卷过期（SURVEY_EXPIRED）]
I --> J[回访状态关闭（FOLLOW_UP_CLOSED）]
J --> H
```

说明：

- 服务完成后，系统自动生成问卷并设置截止日期
    
- 问卷由小程序端展示，客户可在截止日期前填写
    
- 客户提交后，问卷状态变更为 **SURVEY_REPLIED**
    
- 系统需同步更新回访状态为 **已回访（FOLLOW_UP_COMPLETED）**
    
- 若客户在截止日期前未填写，问卷自动变更为 **SURVEY_EXPIRED**
    
- 超时情况下，回访状态自动关闭（FOLLOW_UP_CLOSED）
    
- 问卷流程不影响工单主状态“已完成”，但作为服务质量数据进行记录


### 3.3.12 异常处理流程

系统需支持关键异常处理：

|异常类型|处理方式|
|---|---|
|支付失败|重新发起或关闭|
|支付超时|自动关闭工单|
|库存不足|阻断流程或触发申请|
|物流异常|人工介入处理|
|数据异常|系统校验与修复|
|咨询超时|取消关闭或人工跟进|

```mermaid
flowchart TD

A[任意业务节点] --> B{是否发生异常}

B -->|否| C[继续原流程]

B -->|是| D[记录异常信息]
D --> E[异常分类]

E --> F[物流异常]
E --> G[库存异常]
E --> H[支付异常]
E --> I[数据异常]
E --> J[维修异常]
E --> K[咨询异常]

F --> F1[通知物流/客户]
F1 --> F2[跟踪处理]
F2 --> L[恢复流程]

G --> G1[触发配件申请/库存预警]
G1 --> L

H --> H1[允许重新支付]
H1 --> L

I --> I1[人工修正数据]
I1 --> L

J --> J1[维修中断/重新评估]
J1 --> L

K --> K1[人工跟进或关闭工单]
K1 --> L

L --> M{是否可恢复}

M -->|是| N[返回原流程继续]
M -->|否| O[关闭工单（对应关闭状态）]
```

说明：

- 异常处理流程覆盖主流程各节点
    
- 异常是否可恢复，由具体异常类型和业务规则决定
    
- 无法恢复时需落到明确关闭状态，不允许悬空

# 4. 外部系统协同说明

## 4.1 小程序

主要职责：

- 用户提交售后申请
- 发起支付
- 查询工单状态
- 填写问卷

### 4.1.1 创建工单

```mermaid
sequenceDiagram

```

### 4.1.2 发起支付

```mermaid
sequenceDiagram

    participant M as 小程序
```

```mermaid
sequenceDiagram

    participant A as PS Admin

    A-->>M: 返回当前状态
```

### 4.1.4 问卷调查

```mermaid
sequenceDiagram

participant U as 客户
participant M as 小程序
participant A as PS Admin

%% 问卷生成
A ->> M: 下发问卷信息（工单编号、问卷内容、截止日期）
M ->> U: 展示问卷

%% 用户填写
U ->> M: 填写问卷并提交

%% 数据回传

%% 异常处理
alt 已过期或重复提交
```

### 5.4.4 状态流转规则

小零件申请流程采用标准状态流转：

待审批 → 已通过 → 已出库 → 完成  
　　　　↘ 已拒绝

说明：

- 待审批：申请已提交，等待处理
- 已通过：审批通过，允许出库
- 已出库：库存已扣减，完成发放
- 完成：流程结束
- 已拒绝：审批未通过

---

### 5.4.5 业务规则

1. 申请单必须生成唯一编号  
2. 配件存放位置必须由系统自动带出，不允许手工修改  
3. 当配件类型为“螺钉类”等特殊类型时，产品名称为必填  
4. 数量必须符合数量规则配置  
5. 不同申请主体可配置不同审批策略

---

### 5.4.6 权限与控制

系统支持按申请主体进行权限控制：

|申请主体|是否允许申请|说明|
|---|---|---|
|门店|允许|需审批|
|维修中心|允许|可配置自动通过|
|3PL|允许|需审批|
|仓库|允许|用于补货|

---

### 5.4.7 与库存联动

小零件申请与库存系统需联动：

- 审批通过后方可执行出库
- 出库完成后自动扣减库存
- 所有库存变动需记录库存日志
- 库存不足时需提示或阻断申请

---

### 5.4.8 补充说明

本模块用于统一规范小零件申请流程及库存消耗管理，确保：

- 配件使用可追溯
- 库存变动可管理
- 成本数据可统计
- 多角色协同清晰
