# 1 项目概述

本项目旨在为 Gentle Monster 中国区构建一套覆盖售后全流程的管理系统，支持维修流程管理、配件库存协同及多渠道客户服务。系统将作为客户服务的核心平台，实现流程统一、状态可控及多系统协同，支撑售后业务高效运行。

## 1.1 项目目标  

- 建立统一的售后流程与状态管理体系  
- 提升工单处理效率，优化跨系统协同能力  
- 实现小程序、SAP、物流及支付系统的数据打通  
- 提供清晰的服务进度与用户交互能力  
- 支撑售后数据统计与运营分析

## 1.2 项目范围

统一管理售后服务从申请到完成的各业务环节，规范流程执行，统一状态管理，并实现与各相关系统的数据协同。

### 1.2.1 业务范围

支持多种售后处理模式，包括维修、换货、外部处理及异常处理等典型业务场景。


### 1.2.2 功能范围

系统围绕售后工单生命周期，提供流程管理、业务处理、外部系统协同及消息通知等核心能力。


### 1.2.3 系统集成

系统需与以下外部系统协同：

- 小程序（申请入口、状态查询、支付）
    
- SAP（库存与财务）
    
- 物流系统（寄件与状态同步）

PS Admin 作为流程控制与状态管理中心。


### 1.2.4 不包含范围

本系统不涉及：

- SAP、物流、支付系统的内部逻辑改造
    
- 物流履约能力建设
    
- 支付清结算能力实现
    
- 非售后相关系统建设


### 1.2.5 流程范围边界定义

|流程环节|是否系统内|责任系统|说明|
|---|---|---|---|
|申请|是|小程序 / PS Admin|用户或门店发起|
|受理|是|PS Admin|生成工单|
|收货|否|物流系统|仅同步状态|
|检测|是|PS Admin|录入检测结果|
|判定|是|PS Admin|生成处理方案|
|咨询|是|PS Admin + 小程序|用户确认|
|支付|部分|支付系统|PS Admin仅管理订单|
|维修|部分|PS Admin / 外部|可能由厂家执行|
|出库|否|SAP|PS Admin发起|
|履约|否|物流系统|仅跟踪|
|完成|是|PS Admin|流程闭环|


## 1.3 系统边界与职责划分

PS Admin 作为中国区售后业务的核心系统，负责工单流程驱动、状态管理及业务编排。

在整体售后体系中，PS Admin 作为主控系统，统一接收并处理支付、库存、物流等外部系统返回的业务结果，确保流程按既定规则推进并形成闭环。

外部系统负责具体业务执行（如支付处理、库存操作、物流履约等），不参与流程决策。


### 1.3.1 总体职责定义

|职责类型|说明|
|---|---|
|流程控制|驱动售后全流程（申请→检测→判定→处理→履约→完成）|
|状态管理|维护工单全生命周期状态，作为统一状态来源|
|业务编排|根据判定结果驱动不同处理路径（维修/换货/关闭等）|
|数据整合|汇总支付、库存、物流等外部系统数据|
|异常处理|处理支付失败、库存不足、物流异常等情况|


### 1.3.2 外部系统职责划分

#### 1 支付系统

|系统|职责|
|---|---|
|微信支付|提供支付能力并返回支付结果|
|支付侧（小程序）|发起支付并同步支付结果|
|PS Admin|创建支付订单并根据结果驱动流程|


#### 2 库存 / SAP 系统

|系统|职责|
|---|---|
|SAP|库存管理、出入库执行及主数据维护|
|PS Admin|发起库存请求并记录业务结果|


#### 3 物流系统

|系统|职责|
|---|---|
|物流系统|负责揽收、运输、派送及签收|
|PS Admin|记录运单并跟踪物流状态|


#### 4 小程序（客户端）

|系统|职责|
|---|---|
|小程序|用户操作入口，负责申请、支付及状态查询|
|PS Admin|提供业务接口并驱动流程状态变化|


### 1.3.3 异常处理责任划分

|异常类型|责任系统|是否阻断流程|处理方式|
|---|---|---|---|
|支付失败|小程序支付端|是|返回失败原因，支持重试|
|库存不足|SAP|是|阻断流程，触发库存申请|
|物流异常|物流系统|否|记录异常，支持人工处理|
|接口异常|PS Admin|否|自动重试 + 日志记录|
|数据异常|PS Admin|否|数据校验与修正|


### 1.3.4 设计约束

系统采用状态驱动的流程模型，所有关键操作需具备可追踪能力（日志 + 时间记录），并支持幂等处理及异常重试机制。


### 1.3.5 用户角色定义

|角色|说明|
|---|---|
|客户|发起售后申请并参与流程|
|门店人员|线下受理及代客操作|
|客服人员|处理咨询及工单协同|
|维修人员|执行检测与维修操作|
|系统管理员|系统配置及管理|


### 1.3.6 术语定义

|术语|说明|
|---|---|
|PS Admin|售后管理系统核心后台|
|工单（Service Ticket）|售后流程的核心业务载体|
|3PL|第三方物流或服务提供商|
|SAP|企业ERP系统|
|状态（Status）|工单当前业务阶段|
|动作（Action）|触发状态变更的操作|

- 工单状态变化驱动业务流程推进

## 2.7 设计原则

所有售后业务围绕售后工单展开

### 2 状态驱动原则

业务流程由状态变化驱动

### 3 流程与执行分离原则

PS Admin 负责流程控制，外部系统负责业务执行

### 4 单一数据来源原则

关键数据在指定系统维护

### 5 系统解耦原则

系统之间通过接口交互

### 6 可扩展性原则

支持新增业务与流程扩展

## 3.3 状态模型定义

### 3.3.1 状态模型说明

本系统采用基于状态机（State Machine）的设计，对售后服务单（Service Ticket）的生命周期进行统一管理。

每一个售后工单在任意时刻仅处于一个唯一的业务状态（`status_key`），系统通过状态流转规则控制状态变更的合法性，并在状态变更时触发相应的业务处理与通知动作。

状态模型的核心组成如下：

- 状态定义（`service_status`）：定义所有合法业务状态
    
- 状态流转（`service_status_flow`）：定义状态之间的流转关系
    
- 状态触发机制：定义状态如何被业务事件驱动
    
- 状态与通知关联：状态变化触发通知模板发送


### 3.3.2 状态分类

系统状态按业务阶段划分为以下几类：

|状态分类|说明|
|---|---|
|受理|售后申请已提交并被系统接收|
|收货|产品运输、接收与入库阶段|
|检查|产品进入检测与判定阶段|
|请求|需客户补充资料或补充确认|
|咨询|需与客户进行沟通确认|
|支付|涉及费用支付流程|
|维修|进入维修执行阶段|
|判定|检测后形成处理结论或处理路径|
|出库|产品从维修方或仓库发出|
|通知|物流或门店相关状态通知|
|完成|售后流程已完成|
|关闭|异常或终止类结束状态|

### 3.3.4 状态触发机制

系统状态变更由业务事件驱动，主要来源如下：

|触发来源|说明|
|---|---|
|用户操作|如提交申请、支付、确认方案|
|后台操作|如检测判定、维修操作|
|外部系统回调|小程序支付端、物流、SAP等|
|定时任务|催付、超时关闭等|
|系统自动推进|维修完成、自动出库、完成归档等|

状态变更处理流程如下：

1. 获取当前工单状态（`status_key`）
    
2. 根据“当前状态 + 动作（action_key）”查询流转规则
    
3. 校验是否允许流转
    
4. 更新状态
    
5. 写入状态变更日志
    
6. 触发通知规则（如命中）
    

### 3.3.5 状态与通知关系

系统通过通知触发规则（`notification_trigger_rule`）实现状态与通知模板的解耦。

当发生以下事件时，将触发通知：

- 状态进入（ON_ENTER_STATUS）
    
- 支付提醒（ON_PAYMENT_REMIND_1 / 2 / FINAL）
    
- 出库（ON_OUTBOUND_EXPRESS / STORE）
    
- 完成（ON_COMPLETE）
    
- 满意度调查（ON_SURVEY_SEND）
    


## 3.4 动作模型定义

### 3.4.1 动作模型说明

在状态机模型中，仅定义状态（status）无法完成流程控制，系统必须引入动作（Action）模型，用于驱动状态流转。

系统所有状态变更必须遵循：

```text
当前状态（status_key） + 动作（action_key） → 下一状态（status_key）
```

动作是状态流转的唯一触发入口，用于：

- 控制流程推进
    
- 约束用户/系统操作权限
    
- 统一接口行为
    
- 支撑审计与日志追踪
    

### 3.4.2 动作分类

|动作类型|说明|
|---|---|
|USER_ACTION|用户在小程序或门店触发|
|ADMIN_ACTION|后台人员操作|
|SYSTEM_ACTION|系统自动触发|
|EXTERNAL_ACTION|外部系统回调触发|

### 3.4.4 动作约束规则

系统需满足以下规则：

1. 所有状态变更必须由 action 触发
    
2. action 必须在 `service_status_flow` 中定义
    
3. 不允许直接修改状态（禁止跳状态）
    
4. action 必须记录日志（操作人、时间、来源）
    
5. action 必须支持权限控制
    

### 3.4.5 动作与接口关系

所有接口必须绑定 action_key，例如：

|接口|对应 action|
|---|---|
|创建售后单接口|create_ticket|
|收货入库接口|receive_goods|
|支付回调接口|payment_success|
|维修完成接口|repair_finish|
|出库完成接口|outbound|


## 3.5 状态流转规则

### 3.5.1 流转规则说明

系统通过 `service_status_flow` 表定义状态流转规则，确保：

- 状态变更可控
    
- 流程符合业务逻辑
    
- 不允许非法跳转


每一次状态变更必须满足：

```text
当前状态（status_key） + 动作（action_key） → 下一状态（status_key）
```

|字段|说明|

### 3.5.4 流转控制原则

系统在执行状态流转时必须遵循：

1. 唯一状态原则：一个工单仅有一个当前状态
    
2. 合法流转原则：必须存在于流转表中的路径才允许执行
    
3. 不可逆原则：终态不可再流转
    
4. 事件驱动原则：所有流转必须由明确动作触发
    
5. 日志记录原则：所有状态变更必须记录日志
    

### 3.5.5 与通知的联动规则

当状态发生变更或触发特定事件时：

1. 查询 `notification_trigger_rule`
    
2. 匹配 `status_key` 与 `trigger_event`
    
3. 获取 `template_code`
    
4. 调用通知服务发送消息
    

本系统通过状态机模型实现售后流程的标准化管理，所有业务流程由状态驱动、通过规则控制流转，并通过配置化的通知机制实现与用户的自动化沟通。


## 3.6 详细业务流程

本章流程设计具备以下特点：

- 支持多入口（小程序 / 门店）
    
- 支持多路径（维修 / 换货 / 退回）
    
- 支持子流程解耦（咨询 / 支付 / 配件库存）
    
- 支持跨系统协同
    
- 支持异常处理与流程闭环
    

### 3.6.1 申请流程

用户提交售后申请后生成工单，寄件并被维修中心签收入库后进入检测阶段。  
售后申请支持两种入口：

- 小程序申请
    
- 门店申请
    

```mermaid
flowchart TD

A1[用户在小程序发起申请] --> A2[提交售后信息]
A2 --> A3[生成工单（RECEIVED）]
A3 --> A4[用户填写寄件信息]
A4 --> A5[等待收货（WAIT_RECEIVE）]

B1[客户到门店申请] --> B2[门店创建工单]
B2 --> B3[生成工单（RECEIVED）]
B3 --> B4[门店代寄或客户寄件]
B4 --> B5[等待收货（WAIT_RECEIVE）]

A5 --> C1[维修中心签收]
B5 --> C1
C1 --> C2[入库完成（RECEIVED_WAREHOUSE）]
C2 --> C3[进入检测（INSPECTING）]
```

说明：

- 小程序与门店为两个申请入口，但统一由 PS Admin 生成工单
    
- 进入检测前必须完成签收与入库
    
- 申请流程结束后，工单进入检测主流程
    

### 3.6.2 产品接收流程

```mermaid
flowchart TD

A1[用户/门店寄出产品] --> A2[填写运单号]
A2 --> A3[运输中]
A3 --> A4[维修中心收货]
A4 --> A5[收货登记]
A5 --> A6[入库完成（RECEIVED_WAREHOUSE）]

B1[客户到门店交付产品] --> B2[门店验收]
B2 --> B3[门店登记收货]
B3 --> B4[门店寄往维修中心]
B4 --> A3

A4 --> C1{是否异常}
C1 -->|否| A5
C1 -->|是| C2[记录异常]
C2 --> C3[通知客户/门店]
C3 --> C4[异常处理完成]

A6 --> D1[进入检测（INSPECTING）]
```

说明：

- 快递寄送与门店直收两种收货方式最终汇合为维修中心收货
    
- 产品收货异常需单独记录，并通知相关方
    
- 收货完成后进入检测状态
    

### 3.6.3 检测与判定流程

检测完成后进入判定，根据不同结果流转到不同处理路径。

```mermaid
flowchart TD

A[检测中（INSPECTING）] --> B[检测完成 → 待判定（WAIT_DECISION）]
B --> C{判定结果}

C --> D[无偿维修（WAIT_REPAIR）]
C --> E[有偿维修 → 生成报价]
E --> F[等待支付（WAIT_PAYMENT）]

C --> G[产品更换处理中（EXCHANGE_PROCESSING）]
C --> H[3PL处理 → 进入维修流程（WAIT_REPAIR）]
C --> I[合作厂家维修 → 进入维修流程（WAIT_REPAIR）]

C --> J[无法修理（UNREPAIRABLE）]
C --> K[假货（FAKE_PRODUCT）]

C --> L[需咨询（WAIT_CUSTOMER_CONFIRM）]
```

说明：

- 检测完成后统一进入待判定状态
    
- 判定结果决定是否进入支付、维修、换货、关闭等不同路径
    
- 咨询为判定前后的可插入子流程
    

### 3.6.4 咨询流程

咨询状态默认值为“咨询待机”；当“咨询完成”后，状态自动返回判定节点。

在以下场景需发起咨询：

- 维修方案需客户确认
    
- 费用变更
    
- 他品交换
    
- 合作厂家延迟
    

```mermaid
flowchart TD

A[待客户确认（WAIT_CUSTOMER_CONFIRM）] --> B[发送咨询通知]
B --> C[客户确认/回复]
C --> D{是否确认}

D -->|是| E[返回判定（WAIT_DECISION）]
D -->|否/超时| F[取消工单（CLOSED_CANCELLED）]
```

说明：

- 咨询流程本质上是判定流程中的补充确认子流程
    
- 客户确认后返回待判定状态
    
- 若拒绝或超时，可进入取消关闭
    

### 3.6.5 支付流程

当工单判定为有偿维修场景时，系统进入支付流程。

```mermaid
flowchart TD

A[生成维修费用 WAIT_DECISION] --> B[生成支付订单 WAIT_PAYMENT]
B --> C[小程序查询支付订单 WAIT_PAYMENT]
C --> D[用户查看报价 WAIT_PAYMENT]

D --> E{是否接受报价}

E -->|接受| F[进入待支付 WAIT_PAYMENT]
F --> G[用户发起支付 WAIT_PAYMENT]
G --> H[微信支付完成 WAIT_PAYMENT]

H --> I[小程序接收回调并校验 WAIT_PAYMENT]
I --> J[通知PS Admin WAIT_PAYMENT]

J --> K[更新为已支付 PAID]
K --> L[进入维修流程 WAIT_REPAIR]

E -->|未支付| M[系统催缴提醒 WAIT_PAYMENT]
M --> N{30天内是否支付}

N -->|是| G
N -->|否| O[未付款退回关闭 CLOSED_UNPAID_RETURN]

E -->|拒绝| P[取消关闭 CLOSED_CANCELLED]
```

说明：

- 支付流程是有偿场景的独立子流程
    
- 支付成功后并不直接进入维修中，而是先进入待维修状态
    
- 未支付和拒绝报价对应不同关闭路径
    

### 3.6.6 维修执行流程

若为 “无偿”：判定完成 +D3 后状态自动变更为 “维修进行中”；
若为 “有偿”：付款完成 +D3 后状态自动变更为 “维修进行中”；
若为 “3PL”：付款完成 +D1 后状态自动变更为 “维修进行中”。

```mermaid
flowchart TD

A[判定完成（JUDGED）] --> B{维修类型是否有偿}

B -->|无偿| C[等待至 判定完成 +D3]
B -->|有偿| D{是否已付款（PAID）}
B -->|3PL| E{是否已付款（PAID）}

D -->|是| F[等待至 付款完成 +D3]
D -->|否| Z1[保持待维修（WAIT_REPAIR）]

E -->|是| G[等待至 付款完成 +D1]
E -->|否| Z1

C --> H[状态自动变更为 维修进行中（REPAIRING）]
F --> H
G --> H

H --> I[维修人员领取设备]
I --> J[填写维修开始日期]
J --> K[执行维修处理]

K --> L{维修处理类型}
L -->|维修处理| M[执行维修操作]
L -->|产品更换| N[扣减对应配件]
L -->|他品更换| O[扣减新产品库存]

M --> P[记录维修结果]
N --> P
O --> P

P --> Q[维修完成]
Q --> R[待出库（WAIT_OUTBOUND）]
```

说明：

- 待维修状态用于承接支付完成或无偿判定完成后的统一入口
    
- 满足时间规则后进入维修中
    
- 维修完成后进入待出库，而不是直接出库
    

### 3.6.7 配件与库存处理流程

在维修处理过程中，若涉及配件更换，系统需执行配件选择、库存校验及配件消耗记录等操作。  
该流程作为维修流程中的子流程执行。

1. 配件选择规则  
    维修人员可对系统默认配件进行确认、修改、新增或删除。
    
2. 库存校验规则  
    系统在维修完成前校验配件库存是否满足消耗需求。
    
3. 库存扣减时机  
    配件库存在“维修完成提交时”统一扣减，避免在维修过程中产生库存不一致问题。
    
4. 库存不足处理  
    当库存不足时，系统应支持：
    
    - 触发配件申请流程
        
    - 或生成库存预警
        
5. 数据记录  
    系统需记录：
    
    - 配件消耗明细
        
    - 库存变动记录
        
    - 操作人及时间
        

```mermaid
flowchart TD

A[维修中（REPAIRING）] --> B[选择配件]

B --> C[确认/修改/新增/删除配件]
C --> D[形成配件清单]

D --> E[维修执行]

E --> F[提交维修完成]
F --> G[库存校验]

G --> H{库存是否充足}

H -->|是| I[统一扣减库存]
I --> J[记录库存变动]
J --> K[记录配件消耗明细]
K --> L[记录操作人及时间]
L --> M[维修完成（WAIT_OUTBOUND）]

H -->|否| N{处理方式}

N -->|配件申请| O[触发配件申请流程]
O --> P[等待配件到位]
P --> G

N -->|库存预警| Q[生成库存预警]
Q --> R[人工处理后继续]
R --> G
```

说明：

- 配件与库存流程为维修执行子流程，不单独形成主状态链
    
- 库存扣减统一发生在维修完成提交时
    
- 库存不足时需阻断维修完成提交
    

### 3.6.8 合作厂家 / 3PL 流程

```mermaid
flowchart TD

A[待维修（WAIT_REPAIR）] --> B[发往合作厂家/3PL]
B --> C[维修中（REPAIRING）]
C --> D[厂家维修完成]
D --> E[返回仓库]
E --> F[待出库（WAIT_OUTBOUND）]
```

说明：

- 合作厂家 / 3PL 流程在业务上是外部执行，在系统状态上仍纳入维修主链
    
- 处理完成后统一返回待出库状态
    
- 若涉及费用，仍需先经过支付流程
    

### 3.6.9 无法修理 / 假货流程

```mermaid
flowchart TD

A[无法修理（UNREPAIRABLE）] --> B[通知客户]
B --> C[关闭（CLOSED_UNREPAIRABLE）]

D[假货（FAKE_PRODUCT）] --> E[通知客户]
E --> F[关闭（CLOSED_FAKE）]
```

说明：

- 无法修理与假货均为终止类处理路径
    
- 需先通知客户，再进入对应关闭状态
    
- 不再进入维修、出库和完成流程
    

### 3.6.10 出库与履约流程

```mermaid
flowchart TD

A[待出库（WAIT_OUTBOUND）] --> B[执行出库（OUTBOUND）]

C -->|物流| D[运输中]
D --> E[客户签收（DELIVERED）]
E --> F[服务完成（COMPLETED）]

C -->|门店自提| G[门店到货（STORE_ARRIVED）]
G --> H[通知取件]
H --> I[客户取件（STORE_PICKED_UP）]
I --> F
```

说明：

- 出库后按履约方式区分快递与门店自提
    
- 快递签收与门店取件后均进入已完成
    
- 已完成代表服务业务流程结束
    

### 3.6.11 问卷流程

```mermaid
flowchart TD

A[已完成（COMPLETED）] --> B[系统生成问卷（SURVEY_PENDING）]
B --> C[发送问卷（SURVEY_SENT）]
C --> D[小程序展示问卷]

D --> E{是否在截止日期前填写}

E -->|是| F[提交问卷（SURVEY_REPLIED）]
F --> G[更新回访状态（FOLLOW_UP_COMPLETED）]
G --> H[完成关闭（CLOSED_COMPLETED）]

E -->|否| I[问卷过期（SURVEY_EXPIRED）]
I --> J[回访状态关闭（FOLLOW_UP_CLOSED）]
J --> H
```

说明：

- 服务完成后，系统自动生成问卷并设置截止日期
    
- 问卷由小程序端展示，客户可在截止日期前填写
    
- 客户提交后，问卷状态变更为 **SURVEY_REPLIED**
    
- 系统需同步更新回访状态为 **已回访（FOLLOW_UP_COMPLETED）**
    
- 若客户在截止日期前未填写，问卷自动变更为 **SURVEY_EXPIRED**
    
- 超时情况下，回访状态自动关闭（FOLLOW_UP_CLOSED）
    
- 问卷流程不影响工单主状态“已完成”，但作为服务质量数据进行记录

# 4 跨系统协同与接口设计

## 4.1 协同范围说明

本系统涉及以下系统间协同：

- 小程序（用户入口 / 支付发起）
    
- PS Admin（核心系统）

- 物流系统（顺丰等）

- 同步接口调用
    
- 异步回调 / 通知

## 4.3 支付协同

```mermaid
sequenceDiagram
    participant A as PS Admin
    participant M as 小程序
    participant W as 微信支付

    A->>A: 生成支付订单
    A-->>M: 提供支付订单信息

    M->>A: 查询支付订单
    A-->>M: 返回支付信息

    M->>W: 发起支付
    W-->>M: 支付结果回调

    M->>M: 验签与核单
    M->>A: 通知支付结果

    A->>A: 更新支付状态
```

**PS Admin 负责**

1. **生成待支付订单**  
    当工单判定为需要支付时，由 PS Admin 生成待支付订单，记录订单号、关联工单、应付金额及支付状态。
    
2. **提供待支付订单查询接口**  
    提供接口供微信小程序/支付侧查询待支付订单信息，包括订单号、金额、支付状态、业务单号等。
    
3. **提供支付结果回写接口**  
    提供支付结果回写接口，供小程序支付侧在支付完成并完成验签核单后，将最终支付结果同步至 PS Admin。
    
4. **根据支付结果更新工单状态**  
    在收到支付结果后，更新支付单状态，并根据业务规则同步更新工单状态、支付时间、支付流水号等信息。
    
5. **提供异常补偿处理能力**  
    当支付结果回写失败或状态不一致时，支持通过补偿机制重新获取支付结果并修正业务状态。
    

**微信小程序 / 支付侧负责**

1. **查询待支付订单**  
    通过调用 PS Admin 提供的查询接口，获取当前工单对应的待支付订单信息。
    
2. **调起微信支付**  
    基于待支付订单信息，由支付侧生成支付参数，并在微信小程序中发起支付。
    
3. **接收微信支付回调**  
    接收微信支付返回的异步支付通知，作为支付结果确认的来源。
    
4. **验签、核单**  
    对微信支付回调结果进行验签、金额校验、订单校验，确认支付结果真实有效。
    
5. **通知 PS Admin 最终支付结果**  
    在确认支付成功或失败后，通过支付结果回写接口，将最终支付结果同步给 PS Admin。
    
6. **提供支付结果查询接口**  
    提供支付结果查询接口，供 PS Admin 在支付结果回写失败或状态异常时，通过定时任务主动查询支付状态，进行补单和状态修正。

**支付状态同步原则**

1. **支付最终结果以支付侧验签核单后的结果为准**  
    小程序前端展示结果不能作为最终支付确认依据，必须以支付侧接收到微信支付异步通知并完成验签核单后的结果为准。
    
2. **PS Admin 以支付结果回写作为主同步方式**  
    正常情况下，由支付侧主动将支付结果回写至 PS Admin，作为主链路。
    
3. **PS Admin 以主动查询作为补偿机制**  
    当支付结果回写失败、超时或双方状态不一致时，由 PS Admin 通过定时任务调用支付侧查询接口，获取实际支付状态并完成补单与状态修正。
    
4. **双方均需支持幂等处理**  
    无论是支付结果回写还是补偿查询，同一支付订单多次通知时，系统都应保证重复处理不会造成状态错误或数据重复更新。

## 5.2 售后工单管理

本模块用于在系统中创建和管理维修工单，记录客户信息、产品信息、维修信息、支付信息及物流信息，并驱动售后维修业务流程。
主要业务逻辑规则如下：

1. 维修费用由 SAP 系统返回价格后自动判断为有偿维修。
2. 若维修费用大于 0，则必须完成支付流程。
3. 支付成功后系统自动更新支付状态。
4. 再维修工单自动标记为紧急维修。
5. 出库完成后系统更新维修工单状态为已完成。
6. 若选择快递方式，则必须填写物流单号。

该模块主要实现以下功能：

1. 创建维修工单
2. 记录客户信息
3. 记录产品信息
4. 记录维修受理信息
5. 记录维修信息
6. 管理维修费用及支付信息
7. 管理维修完成后的物流信息
8. 支持附件上传与维修记录管理

|角色|权限|
|---|---|
|售后客服|创建维修工单|
|门店人员|创建维修工单|
|维修工程师|更新维修记录|

## 5.8 问卷与回访管理

本模块用于在服务完成后生成客户满意度问卷，并由小程序端完成问卷展示、填写及回传。  
系统需支持问卷发送、截止日期控制、回访状态管理及结果查询，支撑售后服务评价与运营分析。

1. 服务完成后，系统自动生成满意度问卷
    
2. 小程序端展示待填写问卷并支持客户提交
    
3. 系统记录问卷发送时间、截止日期、回复内容及回复时间
    
4. 支持按问卷截止日期筛选问卷数据
    
5. 支持按回访状态查询问卷处理情况
    
6. 支持关联工单、维修内容与评价内容进行统一查看
    
7. 问卷超期未填写时，系统自动更新为已关闭

# 6 服务时限管理

为保证售后维修服务质量及流程可控性，系统需对售后服务各关键节点设置服务时限（Service Level Agreement，SLA），并通过系统自动监控各阶段处理时间，确保售后流程在规定时间内完成。
SLA规则主要用于：

- 规范售后服务处理时限
- 提高售后处理效率
- 监控流程节点处理时间
- 对超时情况进行预警或自动处理
系统通过记录各节点时间戳，并根据规则自动计算时间差，以实现 SLA 管理。

SLA管理包括以下内容：

1. 售后流程SLA规则
2. 系统自动提醒机制
3. SLA异常处理规则

## 6.1 售后流程时限规则

系统根据售后业务流程，对关键节点设置服务时限要求。

|流程阶段|SLA规则|说明|
|---|---|---|
|售后申请受理|24小时内完成受理|客服确认售后申请并进入受理状态|
|客户寄送产品|无固定 SLA|由客户寄送时间决定|
|产品检测|3个工作日内完成检测|技术人员检测产品并给出判定|
|咨询确认|视客户响应时间|客户确认维修方案|
|报价支付|报价后7天内完成支付|系统自动计算支付截止日期|
|维修处理|判定完成或支付完成后开始维修|根据维修类型自动计算|
|产品出库|维修完成后1个工作日内发货|仓库安排物流|
|服务完成|客户签收后流程结束|系统自动记录完成时间|
|满意度调查|服务完成 +3天|系统自动发送调查|

## 6.2 维修处理时限规则

维修流程的处理时限根据维修类型不同而有所区别。

|维修类型|SLA规则|说明|
|---|---|---|
|无偿维修|判定完成 +3天进入维修|系统自动更新状态|
|有偿维修|支付完成 +3天进入维修|支付完成后进入维修|
|3PL维修|支付完成 +1天进入维修|转第三方维修|
|合作厂家维修|产品移交厂家 +1天进入维修|合作厂家处理|

系统在达到规定时间后自动更新维修状态为 **维修进行中**。

## 6.3 出库与物流时限规则

产品维修完成后，需要在规定时间内完成出库及发货。

|阶段|SLA规则|说明|
|---|---|---|
|维修完成|系统生成预计出库日期|根据维修类型计算|
|普通维修|排除节假日 +10天|默认维修周期|
|紧急维修|排除节假日 +3天|优先处理|
|合作厂家维修|排除节假日 +30天|海外维修|

系统自动生成 **预计出库日期** 并允许管理员根据实际情况调整。

## 6.4 支付时限规则

对于有偿维修订单，系统需要对支付行为进行时间管理。

|阶段|SLA规则|说明|
|---|---|---|
|报价生成|系统生成支付订单|进入待支付状态|
|支付截止时间|报价时间 +7天|默认支付期限|
|支付提醒|每日发送提醒|最多提醒5次|
|最终提醒|第29天再次提醒|防止客户遗漏|
|自动关闭|第30天自动关闭工单|进入退回流程|

## 6.5 时限自动提醒机制

系统根据 SLA 规则自动监控各节点时间，并向相关人员发送提醒。

|触发条件|通知对象|通知方式|
|---|---|---|
|客服未及时受理|客服|系统通知|
|客户未支付维修费用|客户|小程序通知|
|维修即将超期|维修工程师|系统通知|
|出库即将超期|仓库人员|系统通知|
|服务完成|客户|满意度调查通知|

## 6.6 时限异常处理规则

当流程节点超过规定 SLA 时，系统需要记录异常并进行处理。

|异常情况|系统处理|
|---|---|
|客服未及时受理|系统提醒客服|
|客户未支付维修费用|持续提醒客户|
|支付超过30天未完成|系统自动关闭工单|
|维修时间超过预计周期|记录异常并提醒维修负责人|
|出库延迟|提醒仓库人员|

系统需记录所有时限超时事件，并纳入后续运营分析。


# 7 数据模型

## 7.1 设计说明

本章用于定义系统核心业务数据模型，作为数据库设计、接口开发、状态流转控制、页面展示及后续报表分析的统一依据。

数据模型设计遵循以下原则：

- 以售后工单 `service_ticket` 为核心主表
    
- 以状态表 `service_status` 与状态流转表 `service_status_flow` 统一管理流程状态
    
- 所有过程型数据按业务域拆分，避免工单主表字段过度膨胀
    
- 所有关键动作需保留日志与历史记录
    
- 所有对外协同数据应具备可追踪、可重试、可对账能力
    
- 统一采用 `BIGINT` 作为主键
    
- 统一采用 `created_at / updated_at` 作为时间字段命名
    
- 统一采用 `is_deleted` 作为逻辑删除标识（如适用）

## 7.15 关键关系说明

### 7.15.1 主链路关系

- `service_ticket` 为核心主表
    
- `service_ticket_item`、`service_ticket_attachment`、`customer_receive`、`ticket_judgement`、`repair_record`、`payment_record`、`delivery_record`、`consultation`、`service_ticket_exception`、`service_ticket_survey`、`service_ticket_status_log` 均通过 `ticket_id` 关联工单
    
- `customer` 与 `customer_address`、`customer_stats` 通过 `customer_id` 关联
    
- `product_master` 与 `inventory_stock`、`inventory_transaction`、`inventory_request` 通过 `product_id` 关联
    
- `part_master` 与 `part_qty_config`、`part_request_item`、`repair_part`、`repair_order_part_usage`、`part_inventory`、`part_inventory_log` 通过 `part_id` 关联
    
- `store` 与 `store_address`、`store_contact` 通过 `store_id` 关联
    
- `service_status_flow` 通过 `from_status_key / to_status_key` 关联 `service_status`
    
- `notification_trigger_rule` 通过 `status_key / template_code` 关联 `service_status` 与 `notification_template`
    

### 7.15.2 终态定义

以下状态为终态，不允许继续流转：

- `COMPLETED`
    
- `CLOSED_CANCELLED`
    
- `CLOSED_UNPAID_RETURN`
    
- `CLOSED_UNREPAIRABLE`
    
- `CLOSED_FAKE`
    
- `CLOSED_COMPLETED`
    

### 7.15.3 设计约束

- 所有状态变更必须记录到 `service_ticket_status_log`
    
- 所有关键业务修改记录到 `service_ticket_operate_log`
    
- 所有支付记录以 `payment_record` 为准
    
- 所有通知发送记录以 `notification_log` 为准
    
- 所有配件扣减记录以 `repair_order_part_usage` 与 `part_inventory_log` 为准
    
- 所有库存变动记录以 `inventory_transaction` 与 `part_inventory_log` 为准
    
- 所有异常处理需落表到 `service_ticket_exception`
    
- 所有满意度调查发送与回收需落表到 `service_ticket_survey`
